> 노드와 링크가 서로 연결되어 있거나 연결되어 있지 않은 집합체


노드와 간선의 구조 → 그래프의 형태

## ✅네트워크 토폴로지

> 노드와 링크가 어떻게 배치되어 있는지에 대한 방식
> 

![image.png](attachment:5a8bc0d6-b156-42fa-8126-58252232f114:image.png)

### 메시(망) 토폴로지

- 그물망처럼 연결되어 있는 구조
- 장점
    - 한 단말 장치에 장애가 발생해도 경로가 여러개라 계속 사용가능
    - 트래픽 분산 처리
- 단점
    - 노드 추가가 어려움 → 기존의 모든 노드와 연결해야함
    - 구축, 운영 비용이 고가

### 트리 토폴로지

- 트리 형태로 배치한 네트워크 구성
- 노드의 추가, 삭제가 쉬움
    - 부모 - 자식 관계의 구조
    - 해당 계층을 담당하는 중간 허브만 조정하면 됨
    - 즉, 부분이 전체를 방해하지 않는 계층 구조
- 특정 노드에 트래픽이 집중될 때 하위 노드에 영향을 끼칠 수 있음

### 링형 토폴로지

- 각각의 노드가 양 옆의 두 노드와 연결하여 전체적으로 고리처럼 연결된 방식
- 장점
    - 노드가 증가해도 네트워크상 손실이 거의 없음
    - 충돌 가능성이 적고 고장을 쉽게 발견
- 단점
    - 네트워크 구성 변경이 어려움 → 기존 고리를 끊어야 하기 때문
    - 회선에 장애가 발생하면 전체 네트워크에 영향을 크게 끼침

### 성형 토폴로지

- 중앙에 있는 노드에 모두 연결된 네트워크 구성
- 장점
    - 노드를 추가하거나 에러를 탐지하기 쉬움
    - 장애 노드가 중앙 노드가 아닐 경우 다른 노드에 영향을 끼치는 것이 적음
- 단점
    - 중앙 노드에 장애가 발생하면 전체 네트워크를 사용할 수 없음
    - 설치 비용이 고가

### 버스 토폴로지

- 중앙 통신 회선 하나에 여러 개의 노드가 연결되어 공유하는 네트워크
- LAN에 사용됨
- 장점
    - 설치 비용이 적고 신뢰성 우수
- 단점
    - 스푸핑이 가능해서 주의
        - 스푸핑이란
        
        > 스위칭 기능을 마비시켜서 특정 노드에 해당 패킷이 오도록 처리하는 방법
        > 
        
        스푸핑을 적용하면 올바르게 수신부로 가야 할 패킷이 악의적인 노드에 전달

## TCP

### 흐름제어

> 송신 측의 데이터 전송량을 수신축에 따라 조절하는 방법

수신 측의 처리속도보다 송신측의 속도가 빠를 경우 문제 발생
> 

✅ **Stop and Wait**

- 매번 전송한 패킷에 대해 확인 응답을 받아야만 그 다음 패킷을 전송하는 방법

✅ **Sliding Window → GO BACK N** 

![IMG_1970.PNG](attachment:f7393c5f-e8e3-468b-907e-c104210b86be:2f8ca012-889e-46fe-a669-f12e7f9933bc.png)

![IMG_1969.PNG](attachment:1b8ebda2-e7c8-4974-b884-0254cd48ddd3:40bd4daf-4517-41c4-9699-2689209a7bfa.png)

> 오류가 발생하면 오류가 난 지점으로 돌아가서 그 이후의 모든 데이터를 다시 전송
> 

**동작원리**

- 연속 전송
    - 송신측은 수신측의 응답을 일일이 기다리지 않고, 정해진 윈도우 크기 안에서 프레임을 연속적으로 전송
- 누적 응답 : 수신 측은 정상적으로 받은 마지막 프레임에 대해서만 ACK를 보냄
    - 수신 측이 ACK 4를 보내면, 3번 프레임까지는 완벽하게 받은 상태
- 순서 요구
    - 반드시 순서대로 들어오는 프레임만 처리
    - 중간에 하나라도 빠지면 정상적으로 도착한 프레임들도 모두 폐기
- 장점
    - 수신측에서는 순서대로 오지 않으면 버리면 되기 때문에 복잡한 버퍼 관리가 필요 없음
    - 정지-대기 방식에 비해 회선 이용률과 전송 효율이 높음
- 단점
    - 이미 정상적으로 도착한 프레임까지 다시 전부 보내야하므로 대역폭 낭비가 발생

✅ **Selected Repeat**

> 오류가 발생한 프레임만 선택적으로 골라서 재전송하는 방식

대역폭 낭비를 최소화하기 위한 방법
> 

동작 원리

- 개별 응답 : 수신측은 각각의 프레임에 대해 **개별적으로 ACK를 보냄**
- 버퍼의 활용 : 수신측은 순**서가 어긋나게 도착한 프레임들을 버리지 않고 수신 버퍼에 임시로 저장**
- 오류 발생 시 처리
    1. 송신 측이 1, 2, 3, 4, 5 프레임을 전송 → 3번 프레임 유실
    2. 수신 측은 1,2번을 정상 수신 → 4, 5번을 폐기하지 않고 버퍼에 저장 → NAK 3을 보냄
    3. 송신 측이 NAK 3을 받으면, 오류가 난 3번 프레임 단 하나만 다시 전송
    4. 수신 측은 재전송 받은 3번 프레임을 버퍼에 미리 저장해 둔 4, 5번과 함께 올바른 순서로 재정렬하여 상위 계층으로 올려보냄

장점

- 오류가 난 프레임만 전송하여 대역폭을 효율적으로 사용
- 전송 지연이 길거나 오류율이 높은 통신 환경에서 유리

단점

- 수신측에서 순서가 뒤바뀐 프레임을 보관해야 하므로 큰 용량의 버퍼가 필요
- 송신자가 보내는 과거 재전송의 데이터와 새로운 데이터의 시퀀스 번호가 같아 수신자가 이를 구분하지 못함
    - 전체 시퀀스 번호의 개수에 비해 윈도우 크기가 너무 클 때 발생함
    - 윈도우 크기를 전체 시퀀스 번호 개수의 절반 이하로 엄격하게 제

### 혼잡 제어

> 송신자와 수신자 사이의 네트워크 내에서 패킷이 넘쳐나는 것을 방지
> 

✅ **느린 시작**

> 초기 혼잡 윈도우 크기를 1로 시작 → ACK 받을 때마다 윈도우 크기를 2배씩 증가
> 
- 혼잡 현상이 발생하면 사이즈를 1로 떨어뜨림
- 혼잡 현상이 발생했던 window size의 절반까지는 이전처럼 지수 함수 꼴로 창 크기를 증가
- 이후부터는 완만하게 1씩 증가

**✅ 혼잡 회피**

> 윈도우 크기가 특정 임계점에 도달하면, 급격한 지수 증가를 멈추고 윈도우 크기를 선형적 증가
> 
- 

**✅ 빠른 재전송**

> 수신자가 순서에 맞지 않는 패킷을 받으면, 정상적으로 받은 마지막 패킷에 대해 계속 중복 ACK를 보냄
> 
- 중복된 순번의 패킷을 3개 받으면 재전송
- 혼잡을 감지하고 window size를 줄임

**✅ 빠른 회복**

> 혼잡이 발생하여 윈도우 크기를 줄일 때, 완전히 처음으로 가지 않고 절반으로 줄인 상태에서 선형 증가를 시작하는 방식
> 
- 혼잡한 상태가 되면 window size를 1로 줄이지 않고 반으로 줄이고 선형 증가

## 인터넷 계층

> 네트워크 패킷을 IP 주소로 지정된 목적지로 전송하기 위해 사용되는 계층
> 
- IP, ARP, ICMP
- 패킷을 수신해야 할 상대 주소를 지정하여 데이터를 전달
- 비연결형적 특징

> 실질적으로 데이터를 전달하고 장치 간에 신호를 주고받는 규칙을 정하는 계층
> 

## ✅ 캡슐화와 역캡슐화

- 캡슐화
    - 송신 과정에서 헤더를 추가해나가는 과정
- 역캡슐화
    - 캡슐화 과정에서 붙인 헤더를 각 계층에서 확인한 뒤 제거하는 과정
- 계층마다 패킷을 부르는 이름이 다르다
    - 애플리케이션 : 메세지
    - 전송 계층 : 세그먼트
    - 네트워크 : 데이터그램
    - 데이터 : 프레임
    - 물리 : 심볼

# 네트워크 성능 분석 명령어

> 네트워크로부터 발생한 문제점을 확인한 후, 성능 분석을 해야함

이때 필요한 명령어임
> 

## ping

- 대상 노드를 향해 일정 크기의 패킷을 전송하는 명령어
- 노드의 패킷 수신 상태와 도달 시간을 알 수 있음
- 네트워크가 잘 연결되어있는지 확인
- ICMP 프로토콜에서 지원함

## netstat

- 접속되어 있는 서비스들의 네트워크 상태를 표시하는데 사용
- 네트워크 접속, 라우팅 테이블, 네트워크 프로토콜 등 리스트를 보여줌
- 서비스의 포트가 열려 있는지 확인

## nslookup

- DNS에 관련된 내용을 확인하기 위해 쓰는 명령어
- 특정 도메인에 매핑된 IP를 확인하기 위해 사용

## tracert

- 목적지 노드까지 네트워크 경로를 확인할 때 사용하는 명령어
- 어느 구간에서 응답이 느려지는지 확인 가능
