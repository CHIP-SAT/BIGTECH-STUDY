# DNS와 URI/URL

## 도메인 네임과 DNS

✅ 도메인 네임

- [www.example.com](http://www.example.com) 같은 문자열 형태의 호스트 특정 정보
- 호스트의 IP 주소와 대응
- IP 주소에 비해 기억하기 쉬움
- IP 주소가 바뀌더라도 바뀐 IP 주소에 도메인 네임을 다시 대응할 수 있음

✅ 네임 서버 = DNS 서버

- 도메인 네임과 IP 주소를 관리하는 곳
- 전 세계 여러 군데에 위치

- 루트 도메인
    - 마지막 점은 보통 생략
- 최상위 도메인
    - com, net, org , kr
    - jp, cn, us

- 도메인 네임에 대한 관리 체계를 도메인 네임 시스템이라고 부름
- DNS를 사용할 수 있는 애프리케이션 계층 프로토콜

## 자원과 URI/URL

✅ 자원

- 네트워크 상의 메세지를 통해 주고받는 최종 대상
- HTML 파일, 이미지, 동영상 파일, 텍스트 파일

✅ URI

- 웹 상에서 자원을 식별하기 위한 정보
- 자원을 식별하는 통일된 방식
- URN
    - 이름으로 자원을 식별하는 방식
- URL
    - 위치로 자원을 식별하는 방식

✅ URL

![image.png](attachment:9ce25c98-8869-4dc3-84ec-30e47f1d8abf:image.png)

- scheme
    - 자원에 접근하는 방법
    - 일반적으로 사용할 프로토콜이 명시
- authority
    - 호스트를 특정할 수 있는 IP 주소나 도메인 네임이 명시
    - 콜론으로 포트번호
- path
    - 자원이 위치하고 있는 경로
- query
    - URL에 대한 매개변수
    - 특정 단어를 검색, 정렬 방법
    - ?로 시작되는 키-값 형태
- fragment
    - 자원의 일부분, 자원의 한 조각

# HTTP의 특징과 메세지 구조

> 데이터 형식에 구애받지 않고 다양한 애플리케이션 데이터의 송수신을 가능하게함
> 

## HTTP의 특징

1️⃣ 요청 응답 기반 프로토콜

- 클라이언트-서버가 서로 HTTP 요청, 응답 메세지를 주고받는 구조
- 같은 HTTP 메세지라고 하더라도 요청, 응답의 형태는 다르다

2️⃣ 미디어 독립적 프로토콜

- HTML, JPEG, PNG, JSON 등 다양한 종류의 자원을 주고받을 수 있음
- HTTP는 자원의 특성과 무관 → 수단, 즉 인터페이스 역할만 함
- 미디어 타입에 제한 X → 독립적으로 작동이 가능한 미디어 독립적 프로토콜

3️⃣ Stateless 프로토콜

![image.png](attachment:1c72db83-703b-49c3-ae13-1abb28afea69:image.png)

- 서버는 클라이언트 상태를 기억하지 않는다 → 독립적인 요청으로 간주
- 모든 클라이언트의 상태 정보를 유지하는 것은 서버에 큰 부담
    - 서버는 많은 클라이언트와 상호작용을 한다
    - 서버가 여러 대라고 하면 서버끼리 상태 정보를 공유하는 것도 복잡
- 특정 서버에 종속되지 않아 서버의 추가나 대체가 쉬워짐
    - 서버 중 하나에 문제가 생기더라도 쉽게 다른 서버로 대체할 수 있음

4️⃣ 지속 연결 프로토콜

- HTTP 1.0
    - 3-way handshake를 통해 TCP 연결 수립
    - 매번 새롭게 연결을 수립하고 종료해야함
- HTTP 1.1
    - 현재까지도 널리 사용되는 버전
    - 지속 연결 기술 제공
    - 하나의 TCP 연결 상에서 여러 개의 요청-응답을 주고받을 수 있는 기술
- HTTP 2.0
    - 바이너리 데이터 기반 송수신
        - 평문으로 메세지를 주고받는 HTTP 1.1과 달리, 바이너리 데이터를 주고받음
    - 헤더 압축
        - 네트워크 이용 효율을 높임
    - 서버 푸시
        - 미래에 필요할 것으로 예상되는 자원을 미리 전송
    - HTTP 멀티플렉싱 기법
        - 여러 개의 독립적인 스트림을 바탕으로 메세지를 병렬적으로 주고받는 기술
- HTTP 3.0
    - UDP 기반
    - QUIC 프로토콜
    - 송수신 속도가 빠름

# HTTP 헤더

## 요청 메세지에서 주로 사용하는 헤더

✅ Host

- 요청을 보낼 호스트가 명시되는 헤더
- 도메인 네임이나 IP 주소로 표현

✅ User-Agent

- 클라이언트의 프로그램과 관련한 정보가 명시
- 브라우저의 종류, 운영체제 및 아키텍처 정보, 렌더링 엔진
- 클라이언트의 접속 수단을 유추할 수 있

✅ Referer

- 클라이언트가 요청을 보낼 때 머무르던 URL이 명시
- 클라이언트의 유입 경로를 파악할 수 있음

## 응답 메세지에서 주로 사용하는 헤더

✅  Server

- 서버 호스트와 관련된 정보

✅ Allow

- 처리 가능한 HTTP 헤더 목록

✅ Location

- 클라이언트에게 자원의 위치를 알려줄 때
- 리다이렉션이나 새로운 자원이 생성되었을 때 사용

## 둘다 사용

✅ Date

- 메세지가 생성된 날짜, 시각

✅ Content-Length

- 메세지 본문의 바이트 길이

✅ Content-Type, Language, Encoding

- Type
    - 본문에서 사용된 미디어 타입
- Language
    - 특정 언어를 나타내는 언어 코드
    - 특정 국가를 나타내는 국가 코드
    - 두 개를 조합해 어떤 국가에서 사용하는 어떤 언어인지를 알 수 있음
- Encoding
    - 메세지 본문을 압축하거나 변환한 방식
    - gzip, compress, deflate, br

✅ Connection

- HTTP 메세지를 송신하는 호스트가 어떠한 방식의 연결을 원하는지 명시
    - keep-alive : 지속 연결을 희망
    - close :  연결 종료 희
