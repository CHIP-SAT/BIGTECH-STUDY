# TCP와 UDP의 목적과 특징

## 포트를 통한 프로세스 식별

- 웹 브라우저, 게임, 채팅 등 다양한 프로세스가 동작
- 패킷은 최종적으로 프로세스에 도달해야함
- IP 주소와 포트 번호 조합으로, 특정 호스트가 실행하는 특정 프로세스를 식별

## NAT

> 공인 IP 주소와 사설 IP 주소 간 변환이 필요할 때 사용하는 기술

네트워크 내부에서 사설 IP를 사용하는 호스트가 네트워크 외부에 있는 호스트와 패킷을 주고받을 때 사용
> 
- 사설 IP와 공인 IP 주소를 일대일로 변환하면 사설 IP 주소 수만큼 공인 IP 주소가 필요
- 일대일로 대응하지 않고, 사설 IP 주소를 그보다 적은 수의 공인 IP 주소로 변환
- 이때 사용하는 것이 포트
    - 서로 다른 사설 IP 주소가 같은 공인 IP 주소로 변환되더라도 다른 포트로 변환되면 네트워크 내부의 호스트를 특정할 수 있음
- IP 주소쌍, 포트 번호도 함께 고려하는 방법을 NAPT라고 부름

## 신뢰성, 연결형

### TCP

![image.png](attachment:07abdcf7-3af5-4c0d-91ad-b465995662f9:image.png)

- 신뢰할 수 있는 연결형 송수신
- 패킷을 주고받기 전에 연결 수립 과정
- 상태 관리, 흐름 제어, 오류 제어, 혼잡 제어

![image.png](attachment:9452befd-300e-4445-9e4d-372a4cd5d0c6:image.png)

**✅ 순서 번호**

- TCP 패킷의 올바른 송수신 번호를 보장하기 위해 세그먼트 첫 바이트에 매겨진 번호
- 현재 주고받는 TCP 세그먼트가 송수신하고자 하는 데이터의 몇 번째 바이트에 해당하는지 알 수 있음

**✅ 확인 응답 번호**

- 다음으로 수신하길 기대하는 번호
- 올바르게 수신한 순서 번호에 1이 더해진 값
- 승인을 나타내기 위해 ACK 플래그를 추가

**✅ 제어 비트**

- 세그먼트에 대한 부가 정보
- ACK : 세그먼트의 승인
- SYN : 연결을 수립
- FIN : 연결을 종료

**✅ 연결 수립**

3-WAY handshake를 통해 연결을 수립함

1. **SYN 세그먼트 전송**
- SYN 비트: 연결을 수립하기 위한 비트
- 호스트 A의 순서 번호가 포함되어 있음

![image.png](attachment:28ef2e52-3859-42e0-8452-1eca7dc539fa:image.png)

1. **SYN+ACK 세그먼트 전송**
- 호스트 B의 응답
- ACK 비트와 SYN 비트가 1로 설정된 세그먼트를 호스트 A에게 전송
- 세그먼트의 순서 번호에는 호스트 B의 순ㅅ허 번호와 1에서 보낸 확인 응답 번호가 포함
- 

1. **ACK 세그먼트 전송**
- 호스트 A가 비트가 1로 설정된 ACK를 호스트 B에 전송
- 순서 번호에는 호스트 A의 순서 번호 + 2에서 보낸 세그먼트에 대한 확인 응답 번호

## TCP의 오류 흐름 혼잡 제어

패킷의 신뢰성을 보장하기 위해, 3가지 기능을 제공함

- 오류 제어
- 흐름 제어
- 혼잡 제어

**✅ 재전송을 통한 오류 제어**

> TCP는 송수신 과정에서 잘못 전송된 세그먼트가 있을 경우, 재전송하여 오류를 제어

1. 중복된 ACK 세그먼트가 도착했을 때

2. 타임 아웃이 발생했을 때
> 

![image.png](attachment:388cabb7-599a-4781-b282-8cfcc94c9e0d:image.png)

- 송신한 세그먼트의 일부가 전송 중 유실되어 중복으로 ACK 세그먼트를 수신하는 상황

![image.png](attachment:f6f28840-59a0-42cf-9880-c8ac922b7b03:image.png)

- 송신하는 호스트는 재전송 타이머라는 특별한 값을 유지
- 호스트는 세그먼트를 전송할 때마다 재전송 타이머를 시작
- 타이머 카운트다운이 끝난 상황을 타임아웃이라고 하며, 타임아웃까지 ACK 세그먼트를 받지 못하면 세그먼트 전송 과정에 문제가 발생했다고 간주

![image.png](attachment:e3a9adfc-b8b9-465d-8503-a8520025566c:image.png)

**✅ 흐름 제어**

> 수신 호스트가 한 번에 받아 처리할 수 있을 만큼만 전송하는 것

송신 호스트가 수신 호스트의 처리 속도를 고려하여 송수신 속도를 균일하게 맞추는 기능
> 
- 한 번에 받을 수 있는 양은 TCP 수신 버퍼의 크기에 의해 결정
- TCP 헤더에 있는 윈도우 필드를 통해 수신호스트가 한 번에 처리할 수 있는 수신 윈도우 크기를 알 수 있음

**✅ 혼잡 제어**

> 트래픽으로 인해 패킷의 처리 속도가 느려지거나 유실될 수 있는 상황을 제어하는 것
> 
- 송신 호스트가 주체
- 혼잡을 판단하는 방법 : 중복된 ACK, 타임아웃
- 혼잡 윈도우
    - 혼잡 없이 전송할 수 있을 정도의 양의 값만큼
- 리눅스 커널 내에 정의된 값
- RWND, CWND
- 혼잡 제어 알고리즘
    - AIMD
    - 세그먼트를 보내고 그에 대한 응답이 오기까지 혼잡이 감지되지 않으면 혼잡 윈도우를 1씩 선형적으로 증가
    - 혼잡이 감지되면 혼잡 윈도우를 절반으로 떨어뜨리는 동작을 반복하는 알고리즘
    - 혼잡이 감지되면 혼잡 윈도우를 절반으로 떨어뜨리는 알고리즘

## TCP의 종료 : 4-way handshake

1. A→B FIN 세그먼트
- 호스트 A가 FIN 비트 1을 호스트 B에 전송
1. B→A ACK 세그먼트
- 호스트 B가 ACK 세그먼트를 호스트 A에게 전송
1. B→A FIN 세그먼트
- 호스트 B가 A에게 FIN 세그먼트 전송
1. A→B ACK 세그먼트
- 호스트 A가 ACK 세그먼트를 호스트 B에 전송

## TCP의 상태 관리

✅ 연결이 수립되지 않았을 때

- CLOSED
    - 아무런 연결이 없는 상태
- LISTEN
    - 연결 대기 상태 (3-way handshake 첫 단계)

✅ 연결 수립 과정에서 활용하는 상태

- SYN-SENT
    - 액티브 오픈 호스트가 SYN 세그먼트를 보낸 뒤 응답인 SYN+ACK 세그먼트를 기다리는 상태
- SYN-RECEIVED
    - 패시브 오픈 호스트가 SYN+ACK 세그먼트를 보낸 뒤 ACK 세그먼트를 기다리는 상태
- ESTABLISHED
    - 핸드셰이크가 끝난 뒤 데이터를 송수신할 수 있는 상태

✅ 연결 종료 과정에서 활용하는 상태

- FIN-WAIT-1
    - 액티브 클로즈 호스트가 fin 세그먼트로 연결 종료 요청을 보낸 상태
- CLOSE-WAIT
    - fin 세그먼트를 받은 패시브 클로즈 호스트가 그에 대한 응답으로 ACK를 보낸 후 대기
- FIN-WAIT-2
    - FIN-WAIT-1 상태에서 ACK 세그먼트를 받은 상태
- LAST-ACK
    - CLOSE-WAIT 상태에서 FIN 세그먼트를 전송한 뒤 대기
- TIME-WAIT
    - 액티브 클로즈 호스트가 마지막 ACK 전송한 뒤 접어드는 상태

> 액티브 클로즈 호스트는 일정 시간을 기다린 뒤 CLOSED!

### UDP

![image.png](attachment:a1b7750f-a2c4-4579-a249-10ceb40ecf1b:image.png)

> 데이터를 데이터그램 단위로 처리하는 비연결형 프로토콜

비연결형, 신뢰성 없음, 전송 계층
> 

**✅ 핵심 특징**

- 비연결성 : 3-way handshake 불필요
- 비신뢰성 : 데이터가 제대로 전송되었는지 확인 X , 순서 보장이나 재전송도 없음
- 빠른 속도 : 오버헤드가 적어 빠름
- 사용처
    - 실시간 스트리밍
    - 온라인 게임
    - 멀티캐스트/브로드캐스트

**✅ 헤더 구조**

- **Source Port :**송신측 포트 번호
- **Destination Port :** 수신측 포트 번호
- **Length :** 헤더와 데이터를 포함한 전체 길이
- **Checksum :** 데이터 무결성 확인 (오류 검출용 최소한의 장치)

**✅ DNS가 UDP를 사용하는 이유**

- 연결 유지의 불필요
    - DNS 요청은 대개 매우 작음
    - 연결을 맺고 끊는 3-way handshake 과정 자체가 오버헤드
- 빠른 응답성:
    - 수많은 사용자의 도메인 쿼리를 처리해야 하는 DNS 서버 입장에서는 빠른 속도가 생명
- 어플리케이션 계층에서의 제어:
    - 패킷이 유실되더라도 어플리케이션 레벨에서 타임아웃이나 재시도를 처리
    - 전송 계층의 신뢰성이 필수가 아님
