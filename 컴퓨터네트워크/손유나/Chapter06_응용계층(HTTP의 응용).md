## 쿠키

> HTTP의 스테이트리스한 특성을 보완하기 위한 대표적 수단
> 

**서버**

- 쿠키 생성 → 클라이언트에 전송

**클라이언트**

- `이름, 값` 쌍 형태로 저장
- 추후 같은 서버에 요청을 보낼 때, 요청 메시지에 자동으로 쿠키가 포함됨.

<br>

- 서버에서 생성되어 클라이언트 측에 저장되는 `이름, 값` 쌍 형태의 데이터
- 쿠키로 “3일간 보지않기”와 같은 기능 구현 가능
- `domain`과 `path` 속성을 통해 쿠키를 전송할 도메인과 경로 제한 가능
- `Expires`로 명시된 시점이 지나거나 `Max-Age`로 명시된 유효기간이 지나면 해당 쿠키는 삭제되어 전달되지 않음.
    - `Expires`: 쿠키 만료 시점
    - `Max-Age`: 초 단위 유효기간
- 쿠키의 보안 속성
    - `Secure`: HTTPS를 통해서만 쿠키를 송수신하도록 함.
    - `HttpOnly`: 자바스크립트를 통한 쿠키 접근 제한, 오직 HTTP 송수신을 통해서만 쿠키에 접근하도록 함.

## 웹 스토리지

> 쿠키 이외에 클라이언트의 상태를 추측할 수 있는 `키, 값` 쌍 형태의 정보
> 
- 웹 브라우저 내의 저장 공간
- 일반적으로 쿠키보다 더 큰 데이터 저장 가능
- 쿠키와 달리, 정보가 서버로 자동 전송되지 않음.
- 종류
    - 로컬 스토리지: 별도로 삭제하지 않는 한 영구적으로 저장이 가능한 정보
    - 세션 스토리지: 세션이 유지되는 동안(= 브라우저가 열려 있는 동안) 유지되는 정보

## 캐시

> 응답받은 자원의 사본을 임시 저장하여 불필요한 대역폭 낭비와 응답 지연을 방지하는 기술
> 
- 캐시할 데이터의 유효기간을 부여하기 위해 사용하는 헤더
    - `Expires`: 캐시한 데이터의 만료 날짜
    - `Cache Control`의 `max-age`: 캐시하여 사용 가능한 초 단위 시간

**캐시에 유효기간이 부여되는 이유**

- 원본 데이터와 캐시된 사본 데이터 간의 일관성이 깨질 수 있기 때문
- 캐시 신선도: 캐시된 사본 데이터가 서버의 원본 데이터와 얼마나 유사한지의 정도

**캐시 데이터의 유효기간이 지나면 반드시 서버로부터 갱신을 해야할까?** 

- 반드시는 X
- 캐시의 유효기간이 만료되었더라도 캐시된 데이터가 여전히 최신 데이터라면 클라이언트는 굳이 서버로부터 같은 자원을 응답받을 필요 없이, 캐시의 유효기간을 연장하면 됨.
- **캐시의 유효기간이 만료된 경우**
    - 서버에게 원본 자원이 변경된 적이 있는지 질의
        - 날짜 기반 변경 여부 질의
            - `If-Modified-Since` 헤더
                - 헤더에 적힌 특정 시점 이후로 원본 자원에 변경이 있을 때만 변경된 자원을 메시지 본문으로 응답하도록 서버에게 요청
                - 서버가 요청받은 자원이 변경된 경우 → `200(OK)` with 새로운 자원
                - 서버가 요청받은 자원이 변경되지 않은 경우 → `304(Not Modified)`
                - 서버가 요청받은 자원이 삭제된 경우 → `404(Not Found)`
        - 엔티티 태그 기반 변경 여부 질의
            - 엔티티 태그: 자원의 버전을 식별하기 위한 정보
                - 자원이 변경되면 Etag값이 변경됨.
            - `If-None-Match` 헤더
                - 요청할 자원에 대한 Etag 값과 일치하는 Etag가 없다면 변경된 자원으로 응답하도록 서버에게 요청
                - 서버가 요청받은 자원이 변경된 경우 → `200(OK)` with 새로운 자원
                - 서버가 요청받은 자원이 변경되지 않은 경우 → `304(Not Modified)`
                - 서버가 요청받은 자원이 삭제된 경우 → `404(Not Found)`

## 콘텐츠 협상

> 같은 자원에 대해 할 수 있는 여러 표현 중, 클라이언트에게 가장 적합한 자원의 표현을 제공하는 기술
> 
- 같은 URI에 대해서도 다른 자원의 표현이 있을 수 있음
    - ex. 한국어로 표현된 자원, 영어로 표현된 자원
- 여러 표현에 대한 우선순위도 `q`값을 통해 반영 가능
    - ex. `Accept-Language: ko-KR,ko;q=0.9,en-US;q=0.8,en;q=0.7`

## HTTPS · SSL · TLS

- HTTPS: HTTP에 SSL 혹은 TLS라는 프로토콜의 동작이 추가된 프로토콜
- SSL · TLS: 인증과 암호화를 수행하는 프로토콜

**TLS 1.3 기반 HTTPS 메시의 송수신 과정**

1. TCP 쓰리 웨이 핸드셰이크
2. TLS 핸드셰이크
    - 서로에 대한 인증 및 메시지 암호화가 이루어짐.
3. 메시지 송수신

**TLS 핸드셰이크의 핵심**

![image.png](attachment:eeb51509-9de3-44f3-b744-32585d16466b:image.png)

- 암호화 통신을 위한 키 생성·교환
    - TLS에서 활용되는 암호화 알고리즘을 통해 평문을 암호화하거나 반대로 암호문을 복호화하려면 **`키`**라는 정보가 필요
    - 키: 암호화 통신을 수행하는 두 호스트만 알고 있어야 하는 정보
        - TLS 핸드셰이크 과정에서 `ClientHello`와 `ServerHello` 메시지를 주고받으며 생성/교환됨.
            - `ClientHello`메시지: 암호화 이전에 맞춰 봐야 할 정보들을 제시하는 메시지
                - 지원되는 TLS 버전, 사용 가능한 암호화 알고리즘과 해시 함수, 키를 만들기 위해 사용할 클라이언트의 난수 등
            - `ServerHello` 메시지: 제시된 정보들을 선택하는 메시지
                - 선택된 TLS 버전, 암호 스위트 등의 정보, 키를 만들기 위해 사용할 서버의 난수 등
- 인증서 송수신과 검증
    - 인증서: “당신이 통신을 주고받는 상대방은 틀림없이 당신이 의도한 대상이 맞다”라는 사실을 입증하기 위한 정보
        - 인증서는 제 3의 기관이 보장해야 함.
        - CA: 인증서의 발급과 검증, 저장 등의 역할을 수행하는 공인 기관
    - 인증서 및 인증서 검증과 관련한 메시지
        - `Certificate` 메시지: 인증서 내용 포함(ex. 인증서 서명 값 등)
        - `CertificateVerify` 메시지: 인증서의 내용이 올바른지 검증하기 위한 메시지
