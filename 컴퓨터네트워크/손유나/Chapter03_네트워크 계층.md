## IP의 목적

- 주소 지정
    - 네트워크 간의 통신 과정에서 호스트를 특정하는 것
- 단편화
    - 데이터를 여러 IP 패킷으로 올바르게 쪼개어 보내는 것

- 비연결형 통신

## 주소 지정

- 주소 지정은 IP주소를 통해 이루어지며, IP주소는 패킷 헤더를 통해 알 수 있음.
- 하나의 IP주소는 총 4바이트 크기로 구성됨.
- MAC 주소 = 택배 배송 과정의 수신인과 발신인
- IP주소 = 수신 주소와 발신 주소
- 패킷을 올바르게 전송하기 위해서는 **MAC주소**와 **IP주소**가 모두필요함.
    - 패킷의 송수신 과정에서 MAC 주소보다는 **IP 주소가 우선적으로 활용**됨.

**라우터**

> IP 주소를 바탕으로 패킷의 최적 경로를 결정하여 목적지까지 IP 패킷을 전달하는 네트워크 장비
> 

![image.png](attachment:859178ee-f06e-45f8-a5ea-9a269ccad780:image.png)

- 라우터는 IP 패킷을 전달할 최적의 경로를 결정하고 해당 경로로 패킷을 내보낼 수 있어야 함 → **라우팅**
- IPv6: IPv4의 주소가 고갈될 여지가 있어 등장한 프로토콜

## 단편화

- MTU: 최대 전송 단위

![image.png](attachment:5a802ccd-41fb-4365-898d-21d33dcc0c1f:image.png)

- IP 패킷 헤더에서 단편화와 관련된 필드: 식별자, 플래그, 단편화 오프셋

**식별자**

- 특정 패킷이 어떤 데이터에서 쪼개진 패킷인지를 식별하기 위해 사용되는 필드
- 같은 정보에서 쪼개진 패킷들은 같은 식별자를 공유

**플래그**

- 3비트로 구성된 필드
- 첫 번째 비트 → 항상 0
- 두 번째 비트 → DF(IP 단편화를 수행하지 말라)
- 세 번째 비트 → MF(단편화된 패킷이 더 있다)

**단편화 오프셋**

- 패킷의 올바른 순서를 나타내는 데 사용
- 잦은 IP 단편화 → 전송해야 할 패킷의 헤더들이 많아지기 때문에 불필요한 트래픽 증가와 대역폭 낭비를 초래 & 단편화된 패킷을 재조립하는 과정에서 발생하는 부하도 성능 저하로 이어질 수 있음.
    - 따라서 네트워크는 주고받을 수 있는 MTU 크기를 정하고, 해당 크기만큼만 송수신하여 IP 단편화를 회피하는 **경로 MTU 발견** 기술을 지원함.

## 신뢰할 수 없는 통신과 비연결형 통신

- 신뢰할 수 없는 프로토콜: 패킷이 수신지까지 제대로 전송되었다고 보장하지 않는 프로토콜
- 비연결형 프로토콜: 패킷을 주고받기 전에 사전 연결 과정을 거치지 않는 프로토콜

## IP 주소의 구조

- 네트워크 계층은 LAN간의 통신을 가능하게 하고, 이는 IP 주소를 기반으로 이루어짐.
- IP 주소
    - **네트워크 주소(네트워크 ID, 네트워크 식별자)**: 호스트가 속한 네트워크를 특정하기 위해 사용
    - **호스트 주소(호스트 ID, 호스트 식별자)**: 네트워크에 속한 호스트를 특정하기 위해 사용

---

## 클래스풀 주소 체계

- 하나의 IP 주소에서 네트워크 주소를 표현하는 크기와 호스트를 표현하는 크기는 **유동적**임.
- `호스트 주소` > `네트워크 주소` ⇒ 호스트가 할당되지 않은 다수의 IP 주소가 낭비됨.
- `호스트 주소` < `네트워크 주소` ⇒ 호스트가 사용할 IP 주소가 부족
    
    ⇒ 위 고민을 해결하기 위해 생겨난 개념 = **클래스**
    
- 클래스: 네트워크의 크기에 따라 유형별로 IP 주소를 분류하는 기준

**클래스풀 주소 체계**

> 클래스를 바탕으로 IP 주소를 관리하는 주소 체계
> 

![image.png](attachment:5bb107e3-7d29-430b-beb7-dae3507b66c2:image.png)

- 호스트 주소가 전부 `0` 또는 `1`인 IP 주소는 특정 호스트를 지칭하는데 사용 불가
    - 호스트 주소가 전부 `0` → 해당 네트워크 자체를 의미하는 주소
    - 호스트 주소가 전부 `1` → 브로드캐스트를 위한 주소
- 문제점
    - 고정된 네트워크 크기 이외에 다른 크기의 네트워크를 구성할 수 없어 IP 주소가 낭비될 수 있음.

---

## 클래스리스 주소 체계

> 클래스를 이용하지 않고 네트워크와 호스트를 구분하는 방식
> 
- 서브넷 마스크를 이용하여 네트워크와 호스트를 구분
    - 서브넷 마스크: IP 주소상에서 네트워크 주소를 1로 표기하고, 호스트 주소를 0으로 표기한 비트열(서브넷을 구분하는 비트열)
    - 서브넷: IP 주소에서 네트워크 주소로 구분할 수 있는 네트워크의 부분 집합
    - 서브네팅: 서브넷 마스크를 이용해 원하는 크기로 클래스를 더 잘게 쪼개어 사용하는 것
    - ex. 사용자가 192.168.0.1이라는 IP를 가지고 있을 때, 이 주소가 어떤 서브넷(구역)에 속하는지 판별하기 위해 서브넷 마스크라는 기준을 들이댐.
- `서브넷 마스크` 비트 AND 연산 `IP 주소` = IP 주소 내의 네트워크 주소
- CIDR 표기법: `IP주소`/`서브넷 마스크상의 1의 개수` (ex. 192.168.20.3/30)

## **공인 IP 주소**

> 고유한, 인터넷을 비롯한 네트워크 간 통신에서 사용되는 IP 주소
> 
- like 아파트의 도로명 주소

---

## **사설 IP 주소**

> 고유하지 않은, 사설 네트워크에서 사용하기 위한 IP 주소
> 
- 사설 네트워크: 외부 네트워크에 공개되지 않은 네트워크
- like 아파트 내부의 호수(ex. 101호, 202호)
- 사설 IP를 가진 기기(ex. 스마트폰, 노트북)가 인터넷을 하려면 공인 IP가 반드시 필요 ⇒ 공유기가 중간에서 NAT기술 사용
    - NAT: 공유기가 자신의 공인 IP를 발신인 주소로 붙여서 인터넷으로 보냄.
- 사설 IP 주소로 사용하도록 특별히 예약된 IP 주소 공간 ⇒ 인터넷 상에서는 절대 직접 사용 불가
    - 10.0.0.0/8
    - 172.160.0.0/12
    - 192.168.0.0/16
- 사설 IP 주소는 해당 호스트가 속한 사설 네트워크에서만 유효한 주소이므로, 얼마든지 다른 네트워크 상의 사설 IP 주소와 중복될 수 있음.

## 호스트에 IP 주소를 할당하는 방법

**정적 할당**

> 직접 수작업으로 IP 주소를 부여하는 방식
> 
- 필요한 값
    - 정적 IP를 부여하고자 하는 IP 주소
    - 서브넷 마스크
    - 게이트웨이(라우터) 주소
        - 기본 게이트웨이: 호스트가 속한 네트워크의 외부로 나가기 위한 첫 기본 경로
    - DNS 주소
        - 도메인 네임을 토대로 IP 주소를 알아내기 위해 질의하는 서버의 주소
        - 도메인 네임: IP 주소에 대응되는 기억할 수 있는 문자열
        - DNS 서버: `<도메인 네임, IP 주소>` 쌍을 저장하는 서버
            
            ![image.png](attachment:4f28fb14-bb91-4b74-b217-c66bdfe3ef3a:image.png)
            

**동적 할당**

> 프로토콜을 통해 자동으로 IP 주소를 부여하는 방식
> 
- 사용되는 프로토콜 → **DHCP**
- DHCP 서버: 호스트에 할당 가능한 IP 주소 목록을 관리하다가, IP 주소 할당 요청을 받았을 때 IP 주소를 할당해 주는 호스트
    - 라우터가 DHCP 서버 역할 수행
- 동적 IP 주소에는 사용 가능한 기간(임대 기간)이 정해져 있음.
    - IP 주소의 임대 기간이 끝나기 전에 임대 기간 연장 가능 → 임대 갱신
- 동적 IP 주소는 할당 받을 때마다 다른 주소를 받을 수 있음.

## IP가 신뢰할 수 없는 비연결형 프로토콜인 이유

- 신뢰성 높은 송수신을 하려면 패킷에 대한 오류 제어를 수행해야 함. → 더 많은 시간과 대역폭, 부하가 필요 → 성능상 불리하게 작용
- 오류 제어, 연결 수립 및 관리 ↔ 패킷의 빠른 송수신

**IP의 신뢰할 수 없는 비연결형 통신 특징을 보완하는 방법**

- 신뢰할 수 있는 연결형 통신을 지원하는 상위 계층의 프로토콜을 이용
- 네트워크 계층의 프로토콜로 ICMP를 이용

## ICMP

> IP 패킷의 전송 과정에 대한 피드백 메시지(= ICMP 메시지)를 얻기 위해 사용하는 프로토콜
> 

![image.png](attachment:265e753b-c505-4173-9e9b-e0a9570ac63d:image.png)

- ICMP 메시지를 통해 패킷이 상대방에게 어떻게 전송되었는지를 알려줄 수 있음.

**ICMP 메시지**

- **전송 과정에서 발생한 오류 보고**
    - 네트워크 도달 불가
        - 네트워크 장비가 패킷을 전달받았는데, 해당 패킷을 어떤 네트워크로 전송해야 할지 알 수 없을 경우
    - 호스트 도달 불가
    - 프로토콜 도달 불가, 수신지에서 특정 프로토콜을 사용할 수 없음
    - 포트 도달 불가
    - 단편화가 필요하지만 DF가 1로 설정되어 단편화할 수 없음
        - 처리하기에 너무 큰 패킷을 전달받았는데, DF 플래그가 설정되어 있어 단편화가 불가능한 경우
    - TTL 만료
        - TTL 필드가 0이 되어 해당 패킷이 폐기된 경우
        - TTL: 패킷의 수명
            - IP 헤더에는 패킷의 수명을 의미하는 TTL 필드가 있음.
- **네트워크에 대한 진단 정보(네트워크 상의 정보 제공)**
    - Echo 요청
    - Echo 응답

## ARP

> 동일 네트워크 내에 있는 송수신 대상의 IP 주소를 통해 MAC 주소를 알아내는 프로토콜
> 
- **ARP 요청 메시지**
    - 브로드캐스트 메시지
        - 이 메시지를 받은 모든 호스트들은 ARP 요청 메시지에 포함된 IP 주소를 확인 → 자신과 관련이 없는 IP 주소일 경우 무시, 자신의 IP 주소일 경우 ARP 응답 메시지 전송
    - 알고 싶은 MAC 주소에 대응되는 IP 주소 포함
- **ARP 응답 메시지**
    - 응답 메시지를 보내는 호스트의 MAC 주소 포함
- ARP를 활용하는 호스트는 ARP 테이블을 유지
    - ARP 테이블: `<IP 주소, MAC 주소>`의 항목들로 구성된 표 형태의 정보
