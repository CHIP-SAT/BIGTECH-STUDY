## DNS와 도메인 네임

- IP 주소의 한계
    - 특정 호스트의 특징을 나타내기도 어렵고, 호스트의 IP 주소는 언제든 바뀔 수 있음.
    - 해결 → 도메인 네임
- 도메인 네임
    - 문자열 형태의 호스트 특정 정보
    - 호스트의 IP 주소와 대응됨.
- 네임 서버: DNS 서버의 한 종류로, 도메인 네임과 그에 대응하는 IP 주소를 관리하는 서버
- DNS 서버: 도메인 네임을 관리하는 네임 서버로, DNS 프로토콜을 사용하는 서버 전체

**도메인 네임의 계층적 구조**

![image.png](attachment:0d49655d-a9f6-4bad-9894-5321c05ee586:image.png)

- 루트 도메인
- 최상위 도메인(TLD)
- 2단계 도메인(세컨드 레벨 도메인)
- 3단계 도메인
- 전체 주소 도메인 네임(FQDN): “`www.example.com`”처럼 도메인 네임을 모두 포함하는 도메인 네임
    - FQDN을 알면 호스트 식별 가능
    

**도메인 네임을 관리하는 네임 서버의 계층적 구조**

- 도메인 네임 시스템(DNS): 계층적으로 분산되어 있는 도메인 네임에 대한 관리 체계
    - 호스트가 이러한 DNS를 이용할 수 있도록 하는 애플리케이션 계층 프로토콜을 의미하기도 함.

**도메인 네임의 리졸빙 과정**

> 리졸빙: 도메인 이름을 IP 주소로 변환하는 과정
> 

![image.png](attachment:695f3054-44a3-490d-884d-51776d6e7bd6:image.png)

1. 로컬 네임 서버에게 도메인 네임 질의
    - 로컬 네임 서버: 클라이언트가 도메인 네임을 통해 IP 주소를 알아내고자 할 때 가장 먼저 찾게 되는 네임 서버
    - 로컬 네임 서버의 주소는 ISP가 자동으로 할당해줌.
        - 컴퓨터나 스마트폰이 인터넷에 연결될 때, DHCP 프로토콜을 통해 IP 주소뿐만 아니라, 로컬 네임 서버의 주소도 함께 전달 받음.
2. FQDN에 대응하는 IP 주소를 알아낼 때까지 루트 네임 서버 → TLD 네임 서버 → 네임 서버···  등에 걸쳐 질의
    - 루트 네임 서버: 도메인 네임의 루트 도메인을 관장하는 서버
    - TLD 네임 서버: 최상위 도메인을 관장하는 서버
- 문제점
    - 질의 과정이 너무 많이 반복되면 네트워크 내 트래픽이 많아지고, 리졸빙에 지나치게 오랜 시간이 걸림
- 해결
    - 네임 서버들이 기존에 응답받은 결과를 임시로 저장했다가 추후 같은 질의에 활용하는 **DNS 캐시**를 활용

## URI

> 웹 상에서의 자원을 식별하기 위한 정보
> 
- 자원: 두 호스트가 네트워크를 통해 서로 정보를 주고받을 때 송수신하는 대상
- **식별 기준**
    - 이름(URN)
    - 위치(URL)

**URN**

- 이름으로 자원을 식별하는 방식
- 장점
    - 자원에 고유한 이름을 붙이는 이름 기반의 식별자이기 때문에, 자원의 위치와 무관하게 자원 식별 가능

**URL**

- 위치로 자원을 식별하는 방식
- 구조
    
    ![image.png](attachment:703104dd-cda3-4856-aa4b-5f6076470034:image.png)
    
    **scheme**
    
    - 자원에 접근하는 방법
    - 사용할 프로토콜을 명시
    
    **authority**
    
    - 호스트를 특정할 수 있는 IP 주소나 도메인 네임 명시
    - 콜론(:) 뒤에는 포트 번호 명시
    
    **path**
    
    - 자원이 위치하고 있는 경로
    
    **query(쿼리 문자열, 쿼리 파라미터)**
    
    - 자원을 식별하기 위해 추가적인 정보가 필요한 경우 명시
    - 물음표(`?`)로 시작되는 `키=값` 형태의 데이터로 작성
    
    **fragment**
    
    - 자원의 한 조각을 가리키기 위한 정보
    - 일반적으로 HTML 파일과 같은 자원에서 특정 부분을 가리키는 데 사용
- 단점
    - 자원의 위치는 언제든 변할 수 있기 때문에, 자원의 위치가 변하면 URL이 유효하지 않게됨.

## HTTP의 특징

> HTTP의 목적: 데이터의 형식에 구애 받지 않고 애플리케이션의 다양한 자원을 네트워크를 통해 송수신하는 것
> 
- **요청 응답 기반 프로토콜**
    - 클라이언트와 서버가 서로 HTTP 요청 메시지와 HTTP 응답 메시지를 주고받는 구조로 작동
    - HTTP **요청** 메시지의 형태 ≠ HTTP **응답** 메시지의 형태
- **미디어 독립적 프로토콜**
    - 미디어 타입에 무관하게 자원을 주고받는 수단(인터페이스)의 역할만 수행
        - 미디어 타입(MIME 타입): HTTP에서 메시지로 주고받는 자원의 종류
        - 형식: `타입`/`서브타입`
            - 타입: 데이터의 유형(ex. `text`, `image`, `video`)
            - 서브 타입: 주어진 타입에 대한 세부 유형 (ex. `html`,`css`,`jpeg`,`mp4`)
- **스테이트리스 프로토콜**
    - HTTP 요청을 보낸 클라이언트 관련 상태를 기억하지 않음.
- **지속 연결 프로토콜**
    
    HTTP 1.0 이하 → 비지속 연결
    
    ![image.png](attachment:e9bad723-76f6-4a30-a76b-2c5cd68e45bc:image.png)
    
    - 쓰리 웨이 핸드셰이크를 통해 TCP 연결 수립
    - 요청에 대한 응답을 받으면 연결을 종료한는 방식으로 동작
    - 추가적인 요청-응답 메시지를 주고받으려면 매번 새롭게 연결 수립 및 종료 수행
    
    HTTP 1.1, 2.0 → 지속 연결(킵 얼라이브)
    
    ![image.png](attachment:ced2d1b0-4780-4c54-9e5c-3c9cc0f78742:image.png)
    
    - TCP 기반 동작
    - 지속 연결: 하나의 TCP 연결 상에서 여러 개의 요청-응답을 주고받을 수 있는 기술
    - 비지속 연결보다 빠른 속도로 여러 HTTP의 요청·응답 처리 가능

## HTTP가 상태를 유지하지 않는 이유

- 모든 클라이언트의 상태 정보를 유지하는 것은 서버에 큰 부담이 됨.
- 여러 대로 구성된 서버 모두가 클라이언트의 상태 정보를 공유해야 함.

**만약 모든 서버가 클라이언트의 상태 정보를 공유하지 못하면?**

- 특정 클라이언트가 특정 서버에 종속됨. → 어느 한 서버에 문제 발생 시 해당 서버에 종속된 클라이언트는 직전까지의 HTTP 통신 내역을 잃게됨.

**서버가 상태를 유지하지 않고 모든 요청을 독립적으로 처리하면?**

- 특정 클라이언트가 특정 서버에 종속되지 않음. → 서버의 추가나 대체가 용이
- 필요한 경우 언제든, 쉽게 서버를 추가할 수 있어 **`확장성`**을 높일 수 있음.
- 서버 중 하나에 문제가 생기더라도 쉽게 다른 서버로 대체할 수 있어 **`견고성`**을 높일 수 있음.

## HTTP 버전별 특징

- **HTTP `1.1`**
    - 메시지를 평문으로 주고 받음.
    - 콘텐츠 협상 기능 추가
        - 콘텐츠 협상 기능: 같은 URL에 대해 클라이언트와 서버가 서로 “어떤 형식으로 줄게”를 협의하는 메커니즘
- **HTTP `2.0`**
    - HTTP `1.1`의 단점을 보완하고 개선하기 위한 버전
        - HTTP `1.1`의 단점: HOL 블로킹
        - HOL 블로킹: 같은 큐에 대기하며 순차적으로 처리되는 여러 패킷이 있을 때, 첫 번째 패킷의 처리 지연으로 인해 나머지 패킷들의 처리도 모두 지연되는 문제 상황
            
            ![image.png](attachment:280397bf-854a-4d7d-a1bf-3f57c15891dd:image.png)
            
    - 특징
        
        바이너리 데이터 기반 송수신
        
        - 평문이 아닌, 바이너리 데이터를 기반으로 메시지를 주고 받음.
        
        헤더 압축
        
        - 헤더를 압축하여 송수신함으로써 네트워크 이용 효율 상승
        
        서버 푸시
        
        - 클라이언트가 요청하지 않았더라도 미래에 필요할 것으로 예상되는 자원을 미리 전송하는 기능
        
        HTTP 멀티플렉싱 기법
        
        - 여러 개의 독립적인 스트림을 바탕으로 요청-응답 메시지를 병렬적으로 주고받는 기술
        - HOL 블로킹 문제 완화
- **HTTP `3.0`**
    - UDP를 기반으로 구현된 QUIC 프로토콜을 기반으로 동작
    - TCP 속도 < UDP 속도 → 속도 측면에서 큰 개선을 이룸.

## HTTP 메시지 구조

![image.png](attachment:96c655cf-2f1d-4e54-9852-adafd322dd97:image.png)

**시작 라인**

- HTTP 메시지가 요청 메시지인지 응답 메시지인지 구분
    - 요청 메시지 → 시작 라인 = 요청 라인
    - 응답 메시지 → 시작 라인 = 상태 라인
- 요청 라인
    - 형식: `메서드` `요청 대상` `HTTP 버전`
- 상태 라인
    - 형식: `HTTP 버전` `상태 코드` `이유 구문`

**필드 라인**

- HTTP 헤더 명시

## HTTP 메서드

- `GET`, `HEAD`
    - 자원을 조회하는 용도
    - GET: 응답 메시지에 본문이 포함됨.
    - HEAD: 응답 메시지에 본문이 포함되지 않고, 헤더만 받음.
- `POST`
    - 주로 클라이언트가 서버에 새로운 자원을 생성하고자 할 때 사용
- `PUT`, `PATCH`
    - PUT: 덮어쓰기 요청
    - PATCH: 부분적 수정을 요청
- `DELETE`
    - 특정 자원의 삭제를 요청

## HTTP 상태 코드

- 200번대: 성공
    
    ![image.png](attachment:95821cb0-2f40-4f0e-be9f-38c335ac7f0e:6e8f4f77-1de5-455e-ba66-bc89489c0c56.png)
    
- 300번대: 리다이렉션
    
    ![image.png](attachment:84c51610-35aa-4163-bf76-bb47589555e7:image.png)
    
    - 리다이렉션: 클라이언트가 요청한 자원이 다른 곳에 있을 때, 다른 곳으로 요청을 이동시키는 것
        - 영구적 리다이렉션
            - 자원이 완전히 새로운 곳으로 이동하여 경로가 영구적으로 재지정되는 것
            - 요청을 보낸 기존의 URL은 기억할 필요 X
        - 일시적 리다이렉션
            - 자원의 위치가 임시로 변경되었거나 임시로 사용할 URL이 필요한 경우 주로 사용
            - 여전히 요청을 보낸 기존의 URL을 기억해야함.
    
    ![image.png](attachment:c08d2611-8fec-4db0-a137-8c721f1e538f:image.png)
    
    - 서버는 응답 메시지의 Location 헤더를 통해 요청한 자원이 위치한 URL을 안내해줌.
    - 클라이언트는 Location 헤더에 명시된 URL로 재요청
- 400번대: 클라이언트 에러
    
    ![image.png](attachment:dbaad6eb-2d04-4e0b-9836-90d91252fa70:image.png)
    
    - 401(Unauthorized): **인증**을 요구하는 상태 코드
    - 403(Forbidden): **권한(인가)**을 요구하는 상태 코드
    
    **인증 vs 권한**
    
    - 인증: 자신이 누구인지를 증명하는 작업
        - 로그인된 모든 유저(관리자 페이지 접근 불가)
    - 권한: 인증된 주체에게 허용된 작업
        - 관리자(관리자 페이지 접근 가능)
    
    ⇒ 인증이 되었더라도, 권한은 충분하지 않을 수 있음.

## HTTP 주요 헤더

**요청 메시지에서 주로 활용되는 헤더**

- Host
    - 요청을 보낼 호스트 명시
        - 도메인 네임이나 IP 주소
- User-Agent
    - 요청 메시지를 보낸 클라이언트의 프로그램과 관련한 정보 명시
    - 이 헤더를 통해 HTTP 요청 메시지를 보낸 클라이언트의 접속 수단(ex. 웹 브라우저) 유추 가능
- Referer
    - 클라이언트가 요청을 보낼 때 머무르던 URL
    - 이 헤더를 통해 클라이언트의 유입 경로 파악 가능

**응답 메시지에서 주로 활용되는 헤더**

- Server
    - 서버 호스트와 관련된 정보 명시
- Allow
    - 처리 가능한 HTTP 헤더 목록을 알리기 위해 사용
    - `405(Method Not Allowed)`를 응답할 때 함께 사용됨.
- Location
    - 클라이언트에게 자원의 위치를 알려 주기 위해 사용
    - 주로 리다이렉션이 발생했을 때나 새로운 자원이 생성되었을 때 사용

**요청과 응답 메시지 모두에서 활용되는 헤더**

- Date
    - 메시지가 생성된 날짜와 시각에 관련된 정보
- Cotent-Length
    - 메시지 본문의 바이트 단위 크기(길이)
- Content-Type
    - 메시지 본문에서 사용된 미디어 타입
- Content-Language
    - 메시지 본문에서 사용된 자연어
    - `언어코드`-`국가 코드`
        
        ⇒ 어떤 국가에서 사용하는 어떤 언어인지 파악 가능
        
- Content-Encoding
    - 메시지 본문의 압축 또는 변환된 방식
    - 메시지 본문은 여러 번 압축/변환 가능
- Connection
    - 송신 호스트가 어떤 방식의 연결을 원하는지 명시
    - keep-alive: 지속 연결 희망
    - close: 연결 종료 희망
