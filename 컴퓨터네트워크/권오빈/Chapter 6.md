# 6장 응용 계층 -HTTP의 응용

## 쿠키

<aside>

HTTP의 stateless를 보완하기 위한 수단이다.

서버에서 생성되어 클라이언트 측에 저장되는 <이름,값> 쌍 형태의 데이터이다.

![image.png](attachment:92c4ad5f-20de-4501-80eb-25a5909d0a65:image.png)

서버가 클라이언트에게 쿠키를 전송할 때는 `Set-Cookie` 헤더가 활용되고,
클라이언트가 서버에게 쿠키를 전송할 때는 `Cookie` 헤더가 활용된다.

특정 서버로부터 쿠키를 전달받았다면 다음부터 해당 서버에 요청을 보낼 때 전달받은 쿠키를 자동으로 전송한다.

### 세션 방식

![image.png](attachment:09983497-d02a-43db-9ed3-2754a2eb723d:image.png)

사용자가 HTTP 요청의 Body에 인증 정보 (유저이름이나 패스워드 같은 것들) 을 실어 서버로 보낸다. 
서버에서는 해당 인증정보가 유효하면 사용자와 데이터를 식별하는 **Session ID**를 생성한다. 생성된 Session ID는 응답의 **`Set-Cookie`** 헤더에 생성된 세션 아이디를 실어 보내진다.

클라이언트는 해당 세션 아이디를 쿠키에 저장하고, 매 요청마다 세션 아이디를 **`Cookie`** 헤더에 실어 전송한다. 
서버는 전달받은 세션 아이디를 통해 해당 요청의 송신자가 누구인지 식별할 수 있다.

</aside>

## 로컬 스토리지, 세션 스토리지

<aside>

웹 브라우저 내의 저장 공간으로, 일반적으로 쿠키보다 더 큰 데이터를 저장할 수 있다.

로컬 스토리지는 별도로 삭제하지 않는 한 영구적으로 저장이 가능하다.

세션 스토리지는 세션이 유지되는 동안 정보가 유지된다.

</aside>

## 웹 캐시

<aside>

최근 요청된 객체의 사본을 저장한다.
클라이언트에게는 서버 역할, 서버에게는 클라이언트 역할을 한다.
클라이언트의 요구에 대한 응답 시간이 감소한다.

유효기간을 부여하기 위해 **Expires 헤더**를 사용한다.
유효기간을 부여하는 이유는 캐시와 원본 데이터 간의 일관성이 깨질 수 있기 때문이다.

캐시의 유효기간이 만료되었더라도 캐시를 다시 갱신할 필요는 없다.
클라이언트는 **If-Modified-Since 헤더**를 사용하여 이 시점 이후로 서버의 자원이 변경이 된 경우에만 캐시를 갱신한다.
자원의 버전을 식별하는 엔티티 태그를 사용하여 서버의 자원의 변경 여부를 확인하는 방법도 있다.

</aside>

## SSL/TLS와 HTTPS

<aside>

HTTPS는 HTTP에 SSL/TLS가 더해진 프로토콜이다.
**SSL(Secure Sockets Layer)**과 **TLS(Transport Layer Security)**는 모두 인증과 암호화를 수행하는 프로토콜이다.

TLS 1.3 기반의 HTTPS 메시지는 크게 3단계를 거쳐 송수신된다.

1. TCP 3-way 핸드셰이크
2. TLS 핸드셰이크
3. 메시지 송수신

![image.png](attachment:6ef56387-f097-42f1-b298-57e94126c2c0:image.png)

TLS 핸드셰이크를 통해 **암호화 통신을 위한 키를 생성/교환**할 수 있다.
**인증서 송수신과 검증**이 이루어진다.

TLS에서 활용되는 암호화 알고리즘을 통해 평문을 암호화하거나 암호문을 복호화하려면 키가 필요하다.
키는 암호화 통신을 수행하는 두 호스트만 알고 있어야 하는 정보로 TLS 핸드셰이크 과정에서 **ClientHello 메시지, ServerHello 메시지**를 주고받으며 생성/교환된다.

ClientHello 메시지는 사용 가능한 암호화 알고리즘과 해시 함수를 서버에 알려주는 메시지이다. 
아래와 같은 형태의 정보를 **암호 스위트(cipher suite)**라고 한다.

![image.png](attachment:9799721d-dc2d-466c-93f2-bcca6b665d4e:image.png)

서버는 이에 대한 응답으로 ServerHello 메시지를 전송한다.
이 메시지에는 선택된 TLS 버전, 암호화 스위트 등의 정보, 키를 만들기 위해 사용할 서버의 난수 등이 포함되어있다.

**인증서(certificate)**는 ‘통신을 주고받는 상대방이 내가 통신을 의도한 대상이 맞다’라는 사실을 입증하는 정보다.
인증서는 인증 기관을 뜻하는 **CA(Certification AUthority)**에서 발급하고 검증한다.

이후 Finished 메시지와 함께 TLS 핸드셰이크를 마무리한 후 암호화된 데이터를 주고받는다.

</aside>
