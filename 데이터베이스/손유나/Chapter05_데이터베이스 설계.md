## ER 다이어그램(ERD)

> 데이터베이스를 구성하는 요소들의 관계를 나타내는 그림
> 
- 목적: 데이터베이스에 저장되는 엔티티의 구조를 모델링하는 것
- ERD를 활용해 데이터베이스의 구조를 명확하게 정의해 뒀을 때 장점
    - 추후 데이터베이스를 확장하거나 수정할 때 어떤 부분이 영향을 받는지 쉽게 파악 가능 → 유지보수 용이, 개발자 간 원활한 소통 가능
- 비효율적인 데이터베이스 설계의 문제점
    - 데이터의 중복과 불일치 발생 가능성을 높여 데이터베이스에 대한 질의 성능을 저하시킬 수 있음.

## 정규화

> 잠재적인 문제가 발생하지 않도록 테이블의 필드를 구성하고, 필요한 경우 테이블을 나누는 작업
> 
- 정규형: 정규화된 테이블의 형태

**제1 정규형(1NF)**

- 필요충분조건: 모든 속성이 원자 값을 가질 것
- 방법
    - 중복되는 레코드를 감수하고 하나의 테이블로 설계
    - 테이블을 2개로 쪼갬.
    

**제2 정규형(2NF)**

- 필요충분조건: 제1 정규형을 만족함과 동시에, 기본 키가 아닌 모든 필드들이 모든 기본 키에 완전히 종속될 것
- 테이블의 모든 필드가 기본 키에 완전히 종속되어야 하고, 기본키가 여러 필드로 구성될 경우 일부 기본 키에만 종속되어서도 안됨.
    - Y가 X에 종속적
        - 특정 필드 X의 값을 통해 특정 필드 Y의 유일한 값이 결정될 경우
        - X: 결정자, Y: 종속자
- 기본 키가 아닌 필드가 기본 키의 일부에 종속되어 있는 경우 → 부분 함수 종속성
- 기본 키 전체에 완전하게 종속되어 있는 경우 → 완전 함수 종속성
    
    ⇒ 부분 함수 종속성이 없고 완전 함수 종속인 상태
    

**제3 정규형(3NF)**

- 필요충분조건: 제 2정규형을 만족하면서, 기본 키가 아닌 모든 필드가 기본 키에 이행적 종속성이 없을 것
- 이행적 종속 관계
    - A필드가 B필드를 결정하고(A→B) B가 C를 결정한다면(B→C) A도 C를 결정하게 되어(A→C) 종속 관계를 형성
    - 이때 A와 C사이를 이행적 종속 관계라고 함.
- 기본 키가 아닌 나머지 모든 필드들이 간접적으로라도 종속되어서는 안 됨.
- 기본 키가 아닌 나머지 모든 필드는 서로를 유추하거나 결정할 수 없어야 함.

**보이스/코드 정규형(BCNF)**

- 필요충분조건: 제3 정규형을 만족하는 동시에 모든 결정자가 후보 키여야 할 것
    - 결정자: 특정 필드를 식별할 수 있는 필드

## 역정규화

> 검색의 속도를 높이기 위해 분할되어 있는 테이블을 하나로 합치는 작업
> 
- NoSQL은 기본적으로 정규화를 하지 않음.
- 정규화의 문제점
    - 정규화를 거듭하다 도면 테이블에 쪼개지는 경향이 있음.
    - 테이블이 많아지면 조인 연산이 빈번해지고, 다른 테이블을 참조하기 위한 성능상의 비용이 늘어날 수 있음.
