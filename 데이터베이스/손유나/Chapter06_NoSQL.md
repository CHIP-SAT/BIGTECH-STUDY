## NoSQL의 특징

> 레코드를 테이블 형태 이외의 다양한 형태로 저장 가능
> 
- NoSQL의 이점: 확장성, 유연성, 가용성, 성능
    - 높은 부하를 감당하거나 대용량 데이터를 다루는 분산 환경에서 빛을 발함.
- 지향하는 주된 목적: 용이한 스케일  아웃
- NoSQL은 성능 향상을 위해 ACID와 정규화를 엄격히 준수하지 X
    - 따라서 RDBMS에 비해 데이터 무결성과 일관성이 다소 저하될 수 있지만, RDMBS에 비해 큰 성능 향상을 얻을 수 있음.
- 유형
    - 키-값 데이터베이스
    - 도큐먼트 데이터베이스
    - 그래프 데이터베이스
    - 칼럼 패밀리 데이터베이스

---

## 키-값 데이터베이스

> 데이터베이스에 레코드를 키(필드)와 값의 쌍으로 저장하는 데이터베이스
> 
- 키-값 데이터베이스 중에선 레코드를 보조기억장치가 아닌 메모리에 저장해 빠른 데이터베이스 접근 속도를 제공하는 경우도 있음. → Redis, Memcached
    - 인메모리 데이터베이스: 메모리에 저장되는 데이터베이스
- 캐시나 세션 등 비교적 가벼운 정보를 저장

---

## 도큐먼트 데이터베이스(문서지향 데이터 베이스)

> 레코드를 도큐먼트라는 데이터 단위로 저장하고 관리하는 데이터베이스
> 
- 도큐먼트: 정형화되어 있지 않은 NoSQL 레코드의 단위를 통칭하기 위해 사용하는 용어
    - JSON이나 XML과 같은 형식을 도큐먼트로 활용
- 도큐먼트가 모여 컬렉션을 이룸

## 그래프 데이터베이스

> 저장하고자 하는 데이터를 그래프의 노드 형태로 저장하는 데이터베이스
> 
- 노드 간의 연결 관계와 방향 표현 가능
    - SNS의 친구 관계나 교통망과 같이 데이터 간의 관계성이 중요한 레코드를 저장하기 위해 주로 사용

## 칼럼 패밀리 데이터베이스

> 데이터를 행 단위가 아닌 관련 컬럼들의 묶음인 칼럼 패밀리 단위로 저장하고 관리하는 데이터베이스
> 

![image.png](attachment:4f27b3d7-3eb2-4a8c-a7c5-618a1b52ad42:image.png)

- RDBMS와 같이 행과 열이라는 개념이 있고, 로우 키를 통해 특정 행을 식별
- RDMBS와 달리 정규화나 조인을 사용하지 X
- 스키마가 고정되어 있지 않아 자유롭게 열 추가 가능
- 행과 열의 집합이 반드시 엄격한 테이블의 형태를 이루지 X → 열들이 동적으로 변할 수 있고, 거기에 행 데이터들이 대응됨.
- 칼럼 패밀리: 관련 있는 열들을 모은 단위
- 키스페이스: 여러 칼럼 패밀리들을 포괄하는 최상위 단위

## 데이터베이스 분할

> 안정적이고 확장성 높은 데이터베이스 레코드 관리를 위해 테이블을 물리적으로 분할하여 레코드를 저장하는 기술
> 
- **수평적 분할**
    
    : 테이블의 **행**을 기준으로 테이블을 나누어 저장하는 방식
    
    - 수평적 분할이 필요한 사례
        - 테이블에 수많은 레코드가 존재하는 경우
        - 테이블의 레코드를 참조할 때마다 모든 레코드를 한 번에 불러들일 필요가 없는 경우
    - 방법
        - 범위 분할: 레코드 데이터가 가질 수 있는 범위를 정의하고, 해당 범위를 기준으로 테이블을 나누는 방식
        - 목록 분할: 레코드 데이터가 특정 목록(리스트)에 포함된 값을 가질 경우, 해당 레코드를 별도의 테이블루 나누는 방식
        - 해시 분할: 특정 열 데이터에 대한 해시 값을 기준으로 별도의 테이블로 분할하는 방식
        - 키 분할: 키를 기준으로 별도의 테이블로 분할하는 방식
- **수직적 분할**
    
    : 테이블의 **열**을 기준으로 테이블을 나누어 저장하는 방식
    
    - 수직적 분할이 필요한 사례
        - 테이블에 발생하는 트랜잭션 수에 비해 테이블 내에 열이 과도하게 많은 경우
        - 특정 열에 속하는 레코드의 데이터 크기가 다른 열의 레코드에 비해 과도하게 큰 경우
        - 보안 상의 이유로 특정 열을 별개의 테이블로 나누어 저장해야 하는 경우
