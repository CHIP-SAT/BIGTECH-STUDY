## 물리 주소와 논리 주소

- CPU와 프로세스는 메모리의 하드웨어 상 실제 주소인 물리 주소가 아닌, 논리 주소를 이용
    - 논리 주소: 프로세스마다 부여되는 0번지부터 시작하는 주소 체계
- 논리 주소 사용 → 중복되는 물리 주소의 번지 수 X / 중복되는 논리 주소의 번지 수 O
- 메모리 관리 장치(MMU): CPU와 메모리 사이에 위치하여, CPU가 이해하는 논리 주소를 메모리가 이해하는 물리 주소로 변환하는 역할

## 스와핑

> 메모리에 적재된 프로세스들 중 현재 실행되고 있지 않은 프로세스를 임시로 스왑 영역으로 쫓아내고, 그 자리에 다른 프로세스를 적재하여 실행하는 메모리 관리 방식
> 
- 스왑 영역: 보조기억장치의 일부 영역
- 스왑 아웃: 현재 실행되지 않는 프로세스가 메모리에서 스왑 영역으로 옮겨지는 것
- 스왑 인: 스왑 영역에 있는 프로세스가 다시 메모리로 옮겨오는 것

## 연속 메모리 할당과 외부 단편화

> 프로세스에 연속적인 메모리 공간을 할당하는 방식
> 
- 외부 단편화 문제를 내포함.
    - 외부 단편화: 메모리에 할당할 수 있는 전체 공간은 충분하지만, 그 공간들이 작은 조각으로 흩어져 있어 정작 필요한 크기의 프로세스를 적재하지 못하는 현상

## 페이징을 통한 가상 메모리 관리

- 스와핑과 연속 메모리 할당의 문제점
    - 외부 단편화
    - 물리 메모리보다 큰 프로세스 실행 불가
    
    ⇒ 해결방법: 가상 메모리
    
- 가상 메모리: 실행하고자 하는 **프로그램의 일부만** 메모리에 적재해, 실제 메모리보다 더 큰 프로세스를 실행할 수 있도록 만드는 메모리 관리 기법
- 가상 주소 공간: 가상 메모리 기법으로 생성된 논리 주소 공간
- 가상 메모리 관리 기법: 페이징, 세그멘테이션

## 페이징

> 프로세스의 논리 주소 공간을 페이지라는 일정한 단위로 나누고, 물리 주소 공간을 페이지와 동일한 크기의 프레임이라는 일정한 단위로 나눈 뒤 페이지를 프레임에 할당하는 가상 메모리 관리 기법
> 
- 프로세스를 구성하는 페이지는 물리 메모리 내에 불연속적으로 배치될 수 있음.
    
    ![image.png](attachment:cb41df44-21ac-4886-ab16-132057eab6d1:image.png)
    
- 외부 단편화 발생 X
    - 페이지라는 일정한 크기로 잘린 프로세스들을 메모리에 불연속적으로 할당할 수 있다면, 연속 메모리 할당처럼 프로세스 바깥에 빈 공간이 생길 수 없기 때문
- 페이지 단위로 페이지 아웃/페이지 인
    - 프로세스의 일부는 메모리에 적재되고, 일부는 보조기억장치에 적재
        
        ⇒ 프로세스를 실행하기 위해 전체 프로세스가 메모리에 적재될 필요X 
        
- 외부 단편화 문제를 해결할 수 있지만, 내부 단편화 문제 야기
    - 모든 프로세스가 페이지 크기에 딱 맞게 잘리는 것이 아니기 때문
    - 내부 단편화: 프로세스에 할당된 메모리 공간이 필요한 크기보다 커서, 할당된 범위 안에서 사용되지 않고 낭비되는 빈 공간이 발생하는 현상
    
    ![image.png](attachment:8a521f0d-ea20-46f2-8e6a-a288950f748b:image.png)
    
- 요구 페이징(demand paging): 메모리에 프로세스를 적재할 때 처음부터 모든 페이지를 적재하지 않고, 메모리에 필요한 페이지만을 적재하는 기법
- 순수 요구 페이징: 아무런 페이지도 메모리에 적재하지 않은 채 무작정 프로세스를 실행할 수 있는 기법
    - 단, 프로세스의 첫 명령어를 실행하는 순간부터 페이지 폴트 발생

## 페이지 테이블

> 프로세스의 페이지와 실제로 적재된 프레임을 짝지어주는 정보
> 
- 물리 메모리 내에 페이지가 불연속적으로 배치될 경우, CPU가 다음으로 실행할 페이지의 위치를 찾기 어려운 문제를 해결하기 위함.
- 프로세스마다 각자의 페이지 테이블 정보를 가짐.
    - 테이블 엔트리: 페이지 테이블을 구성하고 있는 각각의 행

![image.png](attachment:92213268-28e4-44fc-8584-1ef40dab75ae:image.png)

**페이지 테이블 엔트리에 포함되는 대표적인 정보**

- 페이지 번호, 프레임 번호, 유효 비트, 보호 비트, 참조 비트, 수정 비트
- 유효 비트: 해당 페이지에 접근이 가능한지 여부를 알려 주는 정보
    - 페이지가 메모리에 적재됨 → `1`
    - 페이지가 메모리에 적재되어 있지 않음 → `0`
    - 만일 CPU가 메모리에 적재되지 않은 페이지(유효 비트가 0인 페이지)에 접근하려고 하면 페이지 폴트라는 예외가 발생
- 보호 비트: r,w,x의 조합으로 페이지에 접근할 권한을 제한함으로써 페이지를 보호하기 위해 존재하는 비트
- 참조 비트: CPU가 해당 페이지에 접근한 적이 있는지의 여부를 나타내는 비트
    - 메모리에 적재한 이후에 CPU가 읽거나 쓴 페이지 → `1`
    - 적재한 이후에 한 번도 읽거나 쓴 적이 없는 페이지 → `0`
- 수정 비트(더티 비트): 해당 페이지에 데이터를 쓴 적이 있는지 여부를 알려주는 비트
    - 변경된 적이 있는 페이지 → `1`
        - 페이지를 메모리에서 삭제해야 할 때 페이지의 수정 내역을 보조기억장치에도 반영해 두어야 하므로 보조기억장치에 대한 쓰기 작업이 필요
    - 한 번도 접근한 적 없거나 읽기만 했던 페이지 → `0`

**CPU의 페이지 폴트 처리 과정**

1. 기존 작업 내용 백업
2. 페이지 폴트 처리 루틴 실행
    - 페이지 폴트 처리 루틴: 원하는 페이지를 메모리로 가져와 유효 비트를 1로 변경해주는 작업
3. 페이지 폴트 처리 루틴 실행 후, 메모리에 적재된 페이지 실행

## 페이지 테이블 베이스 레지스터(PTBR)

> 특정 프로세스의 페이지 테이블이 적재된 메모리 상의 위치를 가리키는 특별한 레지스터
> 
- 어떤 프로세스를 실행하려면 이 프로세스의 페이지 테이블이 메모리에 적재된 위치를 알아야 함.
- PTBR은 프로세스마다 가지는 정보이므로 각 PCB에 기록되며, 다른 프로세스로의 문맥 교환이 발생할 때 변경됨.

![image.png](attachment:5e15508a-2aad-457a-bd1b-064d1d5fa7f7:image.png)

**운영체제가 모든 페이지 테이블을 메모리에 적재하는 것을 지양하는 이유**

- 메모리 접근 횟수 증가
    - 페이지 테이블에 접근하기 위해 한 번, 실제 프레임에 접근하기 위해 한 번 ⇒ 총 두 번 메모리에 접근
    - 이를 해결하기 위해 **TLB**라는 **페이지 테이블의 캐시 메모리** 사용
    - TLB 히트: CPU가 접근하려는 논리 주소의 페이지 번호가 TLB에 있을 경우
    - TLB 미스: 페이지 번호가 TLB에 없어, 페이지가 적재된 프레임을 알기 위해 메모리 내의 페이지 테이블에 접근해야하는 경우
    
    ![image.png](attachment:1f0d8efb-4a3f-4778-a0ff-193bbc6b5acc:image.png)
    
- 메모리 용량 많이 차지
    - 프로세스의 크기 증가 → 페이지 테이블 크기 증가
    - 계층적 페이징(다단계 페이지 테이블): 페이지 테이블을 페이징하는 방식
    - 프로세스의 페이지 테이블을 여러 개의 페이지로 자르고, CPU와 가까이 위치한 바깥 쪽에 Outer 페이지 테이블을 하나 더 두어 잘린 페이지 테이블의 페이지들을 가리키게 함.
    - Outer 페이지 테이블만 메모리에 유지하면 됨.
    
    ![image.png](attachment:2e081619-3994-4115-9452-a5bdb443e3e9:image.png)

## 페이징 주소 체계

- 페이징 시스템의 논리 주소 형태: <페이지 번호, 변위>
    - 페이지 번호: 몇 번째 페이지 번호에 접근할 지
    - 변위: 접근하려는 주소가 페이지(프레임) 시작 번지로부터 얼만큼 떨어져 있는지
    - 페이지 테이블을 통해 물리 주소 <프레임 번호, 변위>로 변환됨.

## 페이지 교체 알고리즘

> 메모리에 적재된 페이지 중 보조기억장치로 내보낼 페이지를 선택하는 방법
> 
- 어떤 페이지 교체 알고리즘이 사용되냐에 따라 페이지 폴트의 발생 빈도가 달라짐.
- 스래싱(thrashing): 프로세스가 실제로 실행되는 시간보다 페이징에 더 많은 시간을 소요하여 성능이 저하되는 문제
- **FIFO 페이지 교체 알고리즘**
    - 메모리에 가장 먼저 적재된 페이지부터 스왑 아웃하는 페이지 교체 알고리즘
- **최적 페이지 교체 알고리즘**
    - 앞으로의 사용 빈도가 가장 낮은 페이지를 교체하는 알고리즘
    - 가장 낮은 페이지 폴트율 보장
    - but 실제 구현이 어려움
- **LRU 페이지 교체 알고리즘**
    - 가장 적게 사용한 페이지를 교체하는 알고리즘
