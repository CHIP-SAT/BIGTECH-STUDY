✅ 임계 구역

![image.png](attachment:c81c62d8-d5e8-432b-bd5c-32a77356c2f6:image.png)

> 공유 자원에 접근하는 코드 중 동시에 실행했을 때 문제가 발생할 수 있는 코드
> 
- 다수의 프로세스 혹은 스레드가 동시에 공유 자원에 접근할 경우 문제 발생 가능

✅ Race condition

> 프로세스 혹은 스레드가 동시에 임계 구역의 코드를 실행하여 문제가 발생하는 상황
> 
- 자원의 일관성이 손상될 수 있음
- 2개 이상의 프로세스 혹은 스레드가 임계 영역에 진이하고자 하면 둘 중 하나는 작업이 끝날 때까지 대기
- 경쟁 조건을 방지하면서 임계 구역을 관리하려면 프로세스 또는 스레드를 동기화
    - 실행 순서 제어 : 프로세스 및 스레드를 올바른 순서로 실행
    - 상호 배제 : 동시에 접근해서는 안되는 자원에 하나의 프로세스 및 스레드만 접근

# 멀티스레드 동기화

> 여러 스레드들이 자원을 원활히 공유하도록 하는 기법
> 
- lock 방식
    - 뮤텍스, 스핀락
    - 락을 소유한 스레드만이 임계구역에 진입
    - 락을 소유하지 않은 스레드는 락이 풀릴때까지 대기
- 세마포
    - n개 자원을 사용하려는 m개 멀티스레드의 원활한 관리
    - 자원을 소유하지 못한 스레드는 대기
    - 자원을 다 사용한 스레드는 알림

## 뮤텍스 락

<aside>

> 접근해서는 안 되는 자원에 동시 접근이 불가능하도록 상호 배제를 보장하는 동기화 도구

임계 구역에 접근하고자 한다 → 락을 반드시 획득
임계 구역에서 작업이 끝나면 락을 해제
> 

![image.png](attachment:3df1d25c-6c47-4b99-bfa1-71bc08a463c7:image.png)

- 락 변수
    - true / false 중 한 값
- 대기 큐
    - 락이 열리기를 기다리는 스레드 큐
- 연산
    - lock
        - 락이 열린 상태이면 락을 잠구고 임계구역 진입
        - 락이 잠김 상태이면 현재 스레드를 블록 상태로 만들고 대기 큐에 진입
    - unlock
        - 락을 열린 상태로 변경
        - 대기 큐에서 기다리는 스레드 한 개를 깨움

✅ 특징

- 임계구역의 실행 시간이 짧은 경우 비효율적
    - 락이 잠겨있다 → 컨텍스트 스위칭하여 대기 큐에 대기 → 락이 풀리면 다시 실행
    - 락이 잠겨있는 시간보다 스레드가 잠자고 깨는 데 걸리는 시간이 크면 비효율적
- 단일 CPU를 가진 시스템
- 사용자 응용 프로그램
</aside>

<aside>

## 세마포

<aside>

![image.png](attachment:8120fd70-456b-4e2c-9b6c-e6255b8795df:image.png)

> n개의 공유 자원을 다수 스레드가 공유하여 사용하도록 돕는 기법
> 

![image.png](attachment:fed63489-6eaa-447e-b6aa-7c256dd5a10e:image.png)

![image.png](attachment:4076d906-97cd-4796-a532-788486ecdd17:image.png)

- 변수 S
    - 사용 가능한 공유 자원의 개수를 나타내는 변수 = 임계구역에 진입할 수 있는 프로세스 개수
- 대기 큐
- P/V 연산
    - P연산 = wait() 함수 : 임계 구역 진입 전 호출하는 함수
        - 사용 가능한 공유 자원의 개수를 나타내는 변수 S를 1 감소
        - 변수 S의 값이 0보다 작은지 검사
            - 0 이상 : 임계 구역에 진입
            - 0 미만 : 대기 상태로 전환되어 임계 구역에 진입할 수 없음
    - V 연산 = signal() 함수 : 임계 구역 진입 후 호출하는 함수
        - 변수 S를 1 증가
            - 0보다 크다 → 공유 자원의 개수가 1개 이상 남아있음
            - 0 이하 → 임계 구역에 진입하기 위해 대기하는 프로세스가 있음. 준비상태로 전환

✅ sleep-wait 세마포

- P연산
    - counter- , 대기 큐에서 잠자기
- V연산
    - 사용가능 자원이 있으면 잠자는 스레드 깨우기

✅ busy-wait 세마포

- P연산 : 사용 가능 자원이 생길 때까지 무한 루프 후 자원이 생기면 counter-
- V연산: counter++;

## 조건 변수와 모니터

✅ 조건 변수

> 실행 순서 제어를 위한 동기화 도구

특정 조건 하에 프로세스를 실행/일시 중단함으로써 프로세스나 스레드 실행 순서 제어
> 
- wait
    - 호출한 프로세스 및 스레드의 상태를 대기 상태로 전환하는 함수
- signal
    - wait로 일시 중지된 프로세스 및 스레드의 실행을 재개하는 함수

✅ 모니터

> 공유 자원과 그 공유 자원을 다루는 함수로 구성된 동기화 도구

상호 배제를 위한 동기화 뿐만 아니라 실행 순서 제어를 위한 동기화까지 가능

# 교착 상태 = deadlock

![image.png](attachment:239337b9-cc23-4412-b364-5dcd896763f8:image.png)

> 일어나지 않을 사건을 기다리며 프로세스의 진행이 멈춰 버리는 현상
> 
- 멀티스레드 환경에서 주로 발생
- 커널에서도 발생 가능하지만 거의 발생 X

## 교착 상태의 발생 조건

1️⃣ 상호 배제

- 자원은 한 번에 하나의 스레드에만 할당

2️⃣ 점유와 대기

- 한 프로세스가 자원을 할당받은 상태에서 다른 자원 할당받기를 기다림

3️⃣ 비선점

- 해당 자원을 이용하는 프로세스의 작업이 끝나야만 자원을 이용할 수 있음

4️⃣ 원형 대기

- 프로세스와 프로세스가 요청한 자원이 원의 형태를 이루는 경우

## 교착 상태의 해결 방법

![image.png](attachment:3b02128f-daf4-4068-b63c-1af6c632edf7:image.png)

1️⃣ 교착 상태 예방

- 4가지 필요 조건 중 하나를 충족하지 못하게 하는 방법
- 한 프로세스에 필요한 자원을 몰아주고, 다른 프로세스에 필요한 자원을 몰아주기
- 할당 가능한 모든 자원에 번호를 매기고 오름차순으로 할당

2️⃣ 교착 상태 회피

- 교착 상태가 발생하지 않을 정도로만 조심하면서 자원을 할당하는 방법
- 미래 교착 상태 가능성을 검사하고 발생하지 않을 경우에만 할당

3️⃣ 교착 상태 검출 후 회복

- 교착 상태가 검출되면 프로세스를 자원 선점을 통해 회복 시키기
- 강제 종료

4️⃣ 교착상태 무시

- 교착 상태는 없다고 단정, 아무런 대비책 없음
- 사용자나 관리자가 이상을 느끼면 재실행할 것
