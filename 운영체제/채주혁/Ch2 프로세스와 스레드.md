**✅ 코드 영역**

- 실행 가능한 명령어가 저장되는 공간
- cpu가 읽고 실행할 명령어가 담겨 있기 때문에 쓰기가 금지된 읽기 전용 공간

**✅ 데이터 영역**

- 프로그램이 실행되는 동안 유지할 데이터가 저장되는 공간
- 정적 변수, 전역 변수

**✅ 힙 영역**

- 프로그램을 만드는 사용자가 직접 할당 가능한 저장 공간
    
    
- 메모리 공간을 반환하지 않으면 메모리를 낭비하는 메모리 누수 문제 초래 가능
- 가비지 컬렉션 기능을 제공하여 프로그래밍 언어에서 자체적으로 사용되지 않는 힙 메모리를 해제

**✅ 스택 영역**

- 일시적으로 사용할 값들이 저장되는 공간
- 매개변수, 지역 변수, 함수 복귀 주소 등
- 스택 트레이스 형태의 함수 호출 정보가 저장 가능
    - 특정 시점에 스택 영역에 저장된 함수 호출 정보
    - 문제의 발생 지점을 추적할 수 있어 디버깅에 매우 유리

</aside>

## PCB와 문맥 교환

✅ PCB

<aside>

> 프로세스 제어 블록

프로세스와 관련한 다양한 정보를 내포하는 구조체
> 
- 새로운 프로세스가 메모리에 적재됐을 때 커널 영역에 만들어짐
- 프로세스 실행이 끝나면 폐기
- 프로세스 ID와 프로세스가 실행 과정에서 아용한 레지스터 값, 프로세스 상태, CPU 스케줄링 정보, 메모리 관련 정보, 파일 및 입출력장치 관련 정보가 명시

    - 종료되는 프로세스가 있다면 사용 중이던 자원을 해제하고 PCB도 프로세스 테이블에서 삭제
    - 하나만 존재

</aside>

✅ 문맥 교환

> 기존 프로세스의 문맥을 PCB에 백업하고, PCB에 문맥을 복구하여 새로운 프로세스를 실행하는 것
> 

<aside>

프로세스가 실행된다 → 운영체제에 의해 CPU의 자원을 할당받았다

- 프로세스의 CPU 사용 시간은 타이머 인터럽트에 의해 제한
    - 타이머 인터럽트 : 시간이 끝났음을 알리는 인터럽트
    - 자신의 차례가 되면 정해진 시간만큼 CPU를 사용하고, 자신의 차례를 양보하고 다음 차례가 올 때까지 기다림
- 프로그램 카운터를 비롯한 각종 레지스터 값, 메모리 정보, 실행을 위해 열었던 파일, 사용한 입출력장치 등 중간 정보를 백업해야함
- 문맥 : 프로세스의 수행을 재개하기 위해 기억해야 할 정보
- 운영체제는 해당 프로세스의 PCB에 문맥을 백업하고, 실행할 프로세스의 문맥을 복구
</aside>

<aside>

자주 일어나면 좋을까?

- 프로세스 간에 너무 잦은 문맥 교환이 발생하면 캐시 미스가 발생할 가능성이 높음
- 메모리로부터 실행할 프로세스의 내용을 가져오는 작업이 빈번해짐

        → 큰 오버헤드로 이어질 수 있다

</aside>

## 프로세스의 상태

> 생성 → 준비 → 실행 → 대기 → 종료

프로세스는 여러 상태를 거치며 실행되고, PCB를 통해 프로세스 상태를 인식하고 관리
> 

<aside>

- 생성 상태
    - 프로세스를 생성 중인 상태
    - 메모리에 적재되어 PCB를 할당받은 상태
- 준비 상태
    - 당장 CPU를 할당받아 사용할 수 있지만, 자신의 차례가 아니기 때문에 기다리고 있는 상태
    - 디스 패치
        - CPU 할당을 받으면 실행 상태가 되며, 준비 상태인 프로세스가 실행 상태로 전환되는 것
- 실행 상태
    - CPU를 받아 실행 중인 상태
    - 일정 시간 동안만 CPU를 사용할 수 있음
    - 타이머 인터럽트가 발생하면 준비 상태로 됨
    - 실행 도중 입출력장치를 사용하여 입출력 장치의 작업이 끝날 때까지 기다려야 하면 대기 상태
- 대기 상태
    - 곧장 실행이 불가능한 조건에 놓이는 경우
        - 프로세스가 입출력 작업을 요청하거나 바로 확보할 수 없는 자원을 요청하는 등
    - 실행 가능한 상태가 되면 다시 준비 상태가 되어 CPU 할당을 기다림
- 종료 상태
    - 프로세스가 종료된 상태
    - 운영체제는 PCB와 프로세스가 사용한 메모리를 정리
</aside>

# 멀티프로세스와 멀티스레드

✅ 멀티프로세스

> 동시에 여러 프로세스가 실행되는 것
> 

<aside>

한 프로세스를 구성하는 코드를 동시에 실행하려면?

- 웹 브라우저는 하나의 탭마다 하나의 프로세스로 동작
- 탭 하나를 추가할 때마다 프로세스도 추가되는 모습을 볼 수 있음
- 각기 다른 프로세스들이 자원을 공유하지 않고, 독립저으로 실행됨
    - 같은 작업을 수행하지만 각각의 PID 값이 다르고, 프로세스별로 파일과 입출력장치 등의 자원이 독립적으로 할당
    - 한 프로세스의 실행 과정에서 문제가 발생하더라도 다른 프로세스에 직접적인 영향을 끼치지 않는 경우가 많음
</aside>

✅ 프로세스의 문제점

<aside>

- **프로세스 생성의 큰 오버헤드**
    - 프로세스를 위한 메모리 할당, 부모프로세스로부터 복사
    - PCB 생성, 매핑 테이블 생성

- 프로세스 컨텍스트 스위칭의 큰 오버헤드
    - 현재 CPU 컨텍스트 PCB 저장 + 새 컨텍스트를 PCB에서 CPU로 옮기는 시간
    - CPU가 참고할 매핑 테이블 교체 시간
    - CPU 캐시에 새 프로세스 코드, 데이터 채워지는데 걸리는 시간
1. 프로세스 사이 통신 어려움
    - 프로세스가 다른 프로세스 메로리 접근 불가
    - 커널 메모리, 커널에 의해 마련된 메모리 공간 이용 → 코딩이 어렵고 느림
</aside>

    - 프로세스는 반드시 1개 이상의 스레드로 구성
    - 생성될 때 운영체제에 의해 자동으로 1개의 스레드 생성 : 메인 스레드
- 스레드가 실행할 작업은 함수로 작성

    - 스레드 생성 = TCB 생성
- 스레드 생명과 프로세스 생명
    - 스레드로 만든 함수가 종료하면 스레드 종료 → TCB 제거
    - 프로세스에 속한 모든 스레드가 종료되면 프로세스 종료
</aside>

✅멀티스레드

> 프로세스를 동시에 실행하는 여러 스레드
> 
- 하나의 스레드는 스레드 ID, 프로그램 카운터, 레지스터 값, 스택으로 구성

> 가장 큰 차이 → 자원의 공유 여부
> 
- 서로 다른 프로세스
    - 자원을 공유하지 않아 독립적으로 실행
    - 한 프로세스에 문제가 생겨도 다른 프로세스에는 지장이 없거나 적음
- 같은 프로세스
    - 프로세스의 자원을 공유
    - 동일한 주소 공간의 코드, 데이터, 힙 영역을 공유
    - 한 스레드에 생긴 문제가 프로세스 전체의 문제가 될 수 있음

# 프로세스 간 통신

✅ 프로세스 간 통신

> 프로세스 간에 자원을 공유하고 데이터를 주고 받을 수 있는 방법

기본적으로 자원을 공유하지 않지만 통신을 통해 공유 가능
> 

## 공유 메모리

> 프로세스 간에 공유하는 메모리 영역을 토대로 데이터를 주고받는 방식
> 
- 공유 메모리를 할당하면 프로세스가 메모리 공간을 공유하고 읽고 쓸 수 있음
- 공유 메모리 기반 IPC
    - 자신의 메모리 영역을 읽고 쓰는 것처럼 통신
    - 커널 영역을 거치는 일이 없다
    - 메세지 전달 방식보다 통신 속도가 빠름
    - 서로의 공유 메모리 영역을 동시에 읽고 쓸 경우, 일관성이 훼손

## 메세지 전달

> 프로세스 간에 주고받을 데이터가 커널을 거쳐 송수신되는 방식
> 
- 메세지를 보내는 수단과 받는 수단이 명확하게 구분되어 있음
    - send() : 메세지를 보내는 시스템 콜
    - recv() : 메세지를 받는 시스템 콜
    - 커널의 도움을 적극적으로 받을 수 있으므로 레이스 컨디션, 동기화 문제를 고려하는 일이 상대적으로 적음
    - 주고받는 데이터가 커널을 통해 송수신되므로 공유 메모리 기반 IPC 보다 속도가 느림
- 파이프
    - 단방향 프로세스 간의 통신 도구
    - 단방향으로만 읽고 쓸 수 있는 파이프로 양방향 통신을 할 경우, 읽기용 파이프와 쓰기용 파이프 2개를 이용해 양방향으로 통신하는 경우가 많
- 시그널
    - 프로세스에게 특정 이벤트가 발생했음을 알리는 비동기적인 신호
    - 
- 소켓
- 원격 프로시저
