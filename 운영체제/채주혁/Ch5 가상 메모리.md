# 물리 주소와 논리 주소

✅ 논리 주소

프로세스 내에서 사용하는 주소

- cpu와 프로세스가 사용하는 주소 체계는 논리 주소이므로 중복되는 논리 주소가 존재
- 논리 주소와 물리 주소간의 변환이 이루어져야 함 → MMU로 관리
- 사용자는 물리 주소를 알 수 없음

✅ 물리 주소

물리 메모리= 램에 매겨진 주소

✅ MMU

논리 주소를 물리 주소로 바꾸는 하드웨어 장치

![IMG_2001.PNG](attachment:9c419b50-8e4b-43f5-b1f3-dc36366b9a22:IMG_2001.png)

- CPU에 내장
- CPU가 발생시킨 논리 주소는 MMU에 의해 물리 주소로 바뀌어 물리 메모리에 도달

# 스와핑과 연속 메모리 할당

## 스와핑

> 메모리에 적재된 프로세스는 항상 사용되냐 ? 아니다! 실행 중이 아닌 프로세스도 있다!

프로세스를 임시로 스왑 영역에 쫓아내고, 다른 프로세스를 적재하여 실행하는 메모리 관리 방식
> 
- 스왑 아웃
    - 현재 실행되지 않은 프로세스가 메모리에서 스왑 영역으로 옮겨지는 것
- 스왑 인
    - 스왑 영역에 있는 프로세스가 다시 메모리로 옮겨오는 것

근데 만약 메모리 내 빈 공간이 여러 개라면 어디에 배치해야할까?

## 연속 메모리 할당과 외부 단편화

> 프로세스에 연속적인 메모리 공간을 할당하는 방식
> 

![IMG_2003.PNG](attachment:315b4147-78b7-4da7-8ddd-90093ceeeb57:IMG_2003.png)

✅ 고정 크기 할당

- 메모리를 고정 크기의 파티션으로 나누고, 프로세스당 하나의 파티션 할당
- 파티션의 크기는 모두 같거나 다를 수 있음

✅ 가변 크기 할당

- 메모리를 가변 크기의 파티션으로 나누고 프로세스당 하나의 파티션 할당

✅ 외부 단편화

![IMG_2005.PNG](attachment:5008c49b-39f1-488d-958d-cb8bb6e81638:IMG_2005.png)

빈 공간들이 있지만 그보다 큰 프로세스를 적재하기 어려운 상황

- 메모리에 연속적으로 할당되는 환경에서는 프로세스 실행과 종료가 반복되며 메모리 사이에 빈 공간이 생김
- 빈 공간이지만 그보다 큰 프로세스를 적재하기 어렵고, 메모리 낭비로 이어짐

✅ 내부 단편화

페이지 하나의 크기보다 작은 크기로 발생하게 되는 메모리 낭비

- 모든 프로세스가 페이지 크기에 딱 맞게 잘리는 것이 아님
- 페이지 크기는 10KB인데 프로세스 크기가 107KB → 마지막 페이지는 3KB가 남음

## 페이지 테이블

> 프로세스 페이지와 실제로 적재된 프레임을 짝지어주는 정보
> 
- 페이지 번호와 실제로 적재된 프레임 번호가 대응
- CPU는 페이지 테이블의 페이지 번호만 보고도 적재된 프레임을 찾을 수 있음
- 프로세스마다 각자의 페이지 테이블 정보를 갖고 있으므로 CPU가 서로 다른 프로세스를 실행할 때는 각 프로세스의 페이지 테이블을 참조하여 메모리 접근

✅ 테이블 엔트리

- 페이지 테이블을 구성하고 있는 각각의 행들
- 페이지 번호, 프레임 번호 외에도 다양한 정보들이 포함

✅ 유효 비트

- 해당 페이지에 접근이 가능한지 여부를 알려줌
- 현재 페이지가 메모리, 아니면 보조 기억장치에 적재되어 있는지 알려줌
    - 메모리 : 1
    - 디스크 : 0

✅ 보호 비트

- 페이지 보호 기능을 위해 존재
- rwx : 읽기, 쓰기, 실행
    - 100 : 읽기만 가능
    - 111 : 읽기, 쓰기, 실행 모두 가능

✅ 참조 비트

- CPU가 해당 페이지에 접근한 적이 있는지 여부
- 페이지 적재 후 CPU가 읽거나 쓴 페이지 → 참조 비트 1
- 한 번도 읽거나 쓴 적이 없다 → 페이지 0

✅ 수정 비트

- 해당 페이지에 데이터를 쓴 적이 있는지 여부
- 변경된 적이 있다 → 1
- 변경된 적이 없다 → 0

✅ PTBR 페이지 테이블 베이스 레지스터

특정 프로세스의 페이지 테이블이 적재된 메모리 상의 위치를 가리키는 레지스터

- 각 프로세스의 페이지 테이블은 메모리에 적재
- 프로세스를 실행하려면 이 프로세스의 페이지 테이블이 메모리에 적재된 위치를 알아야 함
- 프로세스마다 가지는 정보이므로 PCB에 기록되며, 문맥 교환이 발생할 때 변경
- 모든 페이지 테이블을 메모리에 적재하는 것은 지양
    - 모든 프로세스의 페이지 테이블을 메모리에 두면 메모리 접근 횟수가 많아짐
    - 메모리 용량을 많이 차지하여 비효율적

# 페이지 교체 알고리즘

> 메모리에 적재된 페이지 중 보조기억장치로 내보낼 페이지를 선택하는 방법
> 
- 메모리에서 사용되지 않을 페이지를 적절하게 보조기억장치로 내보내 페이지 폴트의 발생 빈도가 적도록 설계

✅ FIFO 페이지 교체 알고리즘

- 메모리에 가장 먼저 적재된 페이지부터 스왑 아웃하는 페이지 교체 알고리즘

✅ 최적 페이지 교체 알고리즘

- 사용 빈도가 가장 낮은 페이지를 교체

✅ LRU 페이지 교체 알고리즘

- 가장 적게 사용한 페이지를 교체하는 알고리
