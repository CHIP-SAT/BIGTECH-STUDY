## 보조기억장치

- 대중적으로 활용되는 보조기억장치
    - 하드 디스크 드라이브(하드 디스크, HDD)
        - 자기적인 방식으로 데이터를 읽고 쓰는 보조기억장치
        - 헤드를 통해 플래터에 저장된 데이터에 접근
        
        ![image.png](attachment:6754f984-1210-470a-a1cc-e2c7a86975ca:image.png)
        
    - 플래시 메모리 기반 저장장치
        - 전기적인 방식으로 데이터를 읽고 쓰는 반도체 기반의 저장장치
        - USB 메모리, SD 카드, SSD
- **보조기억장치의 본분**
    - 전원이 꺼져도 데이터를 안전하게 보관하는 것
    - CPU가 필요로 하는 정보를 조금이라도 빠른 성능으로 메모리에 전달하는 것
    
    ⇒ 즉 보조기억장치의 데이터를 안전하고 빠르게 다룰 수 있어야 함 → RAID 기술 사용

## RAID(Redundant Array of Independent Disks)

> 데이터의 안전성 혹은 성능을 확보하기 위해 여러 개의 독립적인 보조기억장치를 마치 하나의 보조기억장치처럼 사용하는 기술
> 
- RAID0
    
    ![image.png](attachment:8d305151-546c-43ad-a2e7-0211608561ec:image.png)
    
    - 데이터를 여러 보조기억장치에서 단순하게 나누어 저장하는 구성 방식
    - 저장되는 데이터가 하드 디스크의 개수만큼 나뉘어 저장됨
    - 장점: 빠른 입출력 속도
        - ex) 데이터 `A1`, `A2`, `A3`, `A4`를 동시에 읽어 들이면 하드 디스크 `1`~`4`에서 동시에, 한 번에 읽어들일 수 있음 ⇒ `4TB`인 저장장치 `1`개를 읽고 쓰는 속도보다 RAID0으로 구성된 `1TB`인 저장장치 `4`개의 속도가 이론상 4배 가량 빠름
    - 단점: 저장된 정보가 안전하지 않음
        - 하드 디스크1에 문제가 생기면 하드 디스크2, 3, 4에 저장된 데이터는 불완전한 데이터가 됨
- RAID1
    
    ![image.png](attachment:4147f65c-4912-4201-99f5-85ed41f60e42:image.png)
    
    - 완전한 복사본을 만들어 저장하는 구성 방식
    - 장점: 복구가 간단하고 안전성이 높음
    - 단점: RAID1에 어떤 데이터를 쓸 때는 원본과 복사본 두 곳에 써야하므로 RAID0보다 쓰기 속도가 느려지고, 복사본이 저장된 크기만큼 사용 가능한 용량이 적어짐.
- RAID4
    
    ![image.png](attachment:074cc1e8-2d3a-4116-9811-dc623d5fc107:image.png)
    
    - 패리티 정보를 저장하는 디스크를 따로 두는 구성 방식
        - 패리티: 오류를 검출할 수 있는 정보
    - 장점: RAID1에 비해 적은 하드 디스크로도 안전하게 데이터 보관 가능
    - 단점: 패리티를 저장하는 장치에 병목 현상 발생
        - 새로운 데이터가 저장될 때마다 패리티를 저장하는 디스크에도 데이터를 쓰게 되므로 하드 디스크4에도 쓰기 작업을 해야 함. → 하드 디스크4가 병목 지점이 될 수 있음
- RAID5
    
    ![image.png](attachment:1d6c3c91-4c27-4129-b7da-a350d395c1a5:image.png)
    
    - 패리티를 분선하여 저장하는 구성 방식
    - RAID4의 단점인 병목 현상 보완
- RAID6
    
    ![image.png](attachment:3e17cb3a-c78c-44a9-9f9c-d7ee75270b00:image.png)
    
    - RAID5의 구성과 동일하지만, 서로 다른 2개의 패리티를 두는 구성 방식
    - 장점: 오류를 검출하고 복구할 수 있는 수단이 2개가 생긴 셈이므로 RAID4나 RAID5에 비해 안전성이 높음
    - 단점: 새로운 정보를 저장할 때마다 함께 저장할 패리티가 2개이므로 RAID5에 비해 쓰기 속도가 느림
- Nested RAID
    
    ![image.png](attachment:ed20ba58-d436-4f26-8e07-c338b6702b82:image.png)
    
    - 여러 RAID 레벨을 혼합한 방식

## 장치 컨트롤러와 장치 드라이버

- CPU가 모든 입출력장치의 작동 방식을 아는 것은 어려움 → 따라서 입출력장치는 CPU와 직접 연결되어 정보를 주고받지 않고, 장치 컨트롤러라는 하드웨어를 통해 연결됨.
- **장치 컨트롤러**: CPU와 입출력장치 사이의 통신을 중개하는 중개자 역할의 하드웨어
    - 장치 컨트롤러의 중개를 통해 입출력 장치가 작동
    - CPU가 장치 컨트롤러와 상호작용함으로써 입출력장치를 작동시키는 것 또한, CPU가 실행하는 프로그램을 통해 이루어짐.
- **장치 드라이버**: 장치 컨트롤러의 동작을 알고, 장치 컨트롤러가 컴퓨터 내부와 정보를 주고받을 수 있도록 하는 프로그램
    - ex) 윈도우, 맥 OS와 같은 운영체제에 포함된 장치드라이버 → 별도의 설치 없이 입출력장치 사용 가능

## 프로그램 입출력

> 프로그램 속 명령어로 입출력 작업을 수행하는 방법
> 
- 프로그램 입출력 방식은 입출력장치의 주소를 식별하는 방식에 따라 나뉨
    - 고립형 입출력
        - 입출력장치에 접근하는 주소와 메모리에 접근하는 주소를 별도의 주소 공간으로 간주하는 방식
        - 입출력장치만을 위한 주소 공간에 접근하려면 별도의 입출력 명령어 필요
    - 메모리 맵 입출력
        - 입출력장치에 접근하는 주소 공간과 메모리에 접근하는 주소 공간을 구분하지 않고, 메모리에 부여된 주소 공간 일부를 입출력장치를 식별하기 위한 주소 공간으로 사용하는 방식
        - 입출력 전용 명령어 필요 X → 메모리에 접근하는 명령어로 입출력 가능
    
    ![image.png](attachment:18ce7cdd-d963-4051-9186-7b9faaec2ef6:image.png)
    
<br/>

**인터럽트 기반 입출력 - 다중 인터럽트**

- 플래그 레지스터 속 인터럽트 비트 활성화→ 우선순위가 높은 인터럽트부터 먼저 처리
- 플래그 레지스터 속 인터럽트 비트 비활성화 → 인터럽트 무시 (단, NMI 인터럽트가 발생한 경우에는 무시할 수 없음)

![image.png](attachment:ed7361da-8f84-4263-ae9a-ea4f46230174:image.png)

- 다중 인터럽트를 처리하기 위해선 프로그래머블 인터럽트 컨트롤러(PIC)라는 하드웨어 사용
    - PIC: 여러 장치 컨트롤러에 연결되어 있어 장치 컨트롤러에서 보낸 하드웨어 인터럽트 요청들의 우선순위를 판별한 뒤, CPU에게 지금 처리해야 할 하드웨어 인터럽트가 무엇인지 알려주는 장치

<br/>

- 프로그램 기반 입출력과 인터럽트 기반 입출력은 CPU가 입출력장치와 메모리 간의 데이터 이동을 주도해야 하며, 이동하는 데이터들도 반드시 CPU를 거침
- 입출력장치의 데이터를 메모리에 저장하는 경우
    1. CPU가 장치 컨트롤러로부터 데이터를 하나씩 읽어 레지스터에 적재
    2. 적재한 데이터를 하나씩 메모리에 저장
    
    ![image.png](attachment:b3791b12-2425-4b54-823b-43930c6c4cc7:image.png)
    
    ⇒ 입출력장치와 메모리 사이에 전송되는 모든 데이터가 반드시 CPU를 거치면 CPU의 부담이 커짐.
    
<br/>

**DMA(Direct Memory Access) 입출력**

- CPU를 거치지 않고도 입출력장치와 메모리가 상호작용할 수 있는 입출력 방식
- DMA 입출력을 위해선 시스템 버스에 연결된 DMA 컨트롤러라는 하드웨어 필요
- DMA 컨트롤러는 **시스템 버스**에 연결되고, 입출력장치들의 장치 컨트롤러들은 **입출력 버스(**입출력장치 컨트롤러의 전용 버스)와 연결됨.
- DMA 입출력 과정
    
    ![image.png](attachment:b7353cd5-a125-47ad-b4bb-a5f81f625579:image.png)
    
    1. CPU가 DMA 컨트롤러에게 입출력장치의 주소, 수행할 연산, 연산할 메모리 주소 등의 정보와 함께 입출력 작업을 명령
    2. DMA 컨트롤러가 CPU 대신 장치 컨트롤러와 상호작용하며 입출력 작업을 수행
        - DMA 컨트롤러는 필요한 경우 메모리에 직접 접근하여 정보를 읽거나 씀. ⇒ 입출력장치와 메모리 사이에 주고받을 데이터가 CPU를 거치지 않음.
    3. DMA 컨트롤러는 입출력 작업이 끝나면 CPU에게 인터럽트를 걸어 작업이 끝났음을 알림

## 사이클 스틸링

> DMA의 시스템 버스 사용
> 
- 버스는 공용 자원이므로 두 장치가 동시에 하나의 버스 이용 불가
- CPU가 시스템 버스 사용 ⇒ DMA 컨트롤러는 시스템 버스 사용 불가
- DMA 컨트롤러가 시스템 버스 사용 ⇒ CPU는 시스템 버스 사용 불가
- DMA 컨트롤러는 CPU가 시스템 버스를 사용하지 않을 때마다 조금씩 사용하거나, CPU가 시스템 버스 사용을 양보

## GPU

> 그래픽 처리 장치로, 화면에 그림을 그리는 등 대량의 그래픽 연산을 위해 탄생한 장치
> 
- GPGPU(General-Purpose computing on Graphics Processing Units)
    - 범용적인 목적의 GPU 사용 기술
    - 그래픽 연산뿐만 아니라, CPU의 연산 범위까지 확대되어 딥러닝 연산, 가상화폐 채굴 등 다양한 분야에 대한 연산이 가능해진 것
- GPU는 수백 개에서 수천 개의 코어를 포함하므로 병렬 처리에 용이
    - 어떠한 크고 복잡한 문제를 쉽고 간단한 여러 문제로 쪼갠 뒤, 쉽고 간단한 문제를 처리할 수 있는 수단을 동시에 동원하여 빠르게 문제를 해결하는 방식
    
    ![image.png](attachment:7c8cf25e-57d0-4c42-b6a2-2aa14833829f:image.png)
    
- GPU는 자체적으로 캐시 메모리를 갖춤
- GPU 개별 코어는 CPU에 비해 느릴 뿐만 아니라 크기도 작고, CPU처럼 복잡한 기능을 제공하지 X
- GPU와 CPU는 용도가 다름
    - GPU의 목표: 메모리의 대역폭을 넓혀 최대한 많은 코어가 많은 작업을 받아 처리하는 것(산술 연산과 같이 단순한 연산을 빠르게, 병렬적으로 수행하기 위한 장치)
    - CPU의 목표: 메모리 접근을 가급적 최소화하는 것(범용적인 연산을 수행하기 위한 장치)
