
# RAM

- 휘발성 저장장치
    - 전원을 끄면 저장하고 있던 데이터와 명령어가 날아감
- 비휘발성 저장장치
    - 전원이 꺼져도 저장된 내용이 유지
- RAM의 용량이 작으면 보조기억장치로부터 실행할 프로그램을 가지고 오는 일이 잦아져 실행 시간이 길어짐

✅ DRAM

- 저장된 데이터가 동적으로 변하는 특성
- 시간이 지나면 저장된 데이터가 점차 사라짐
- 소비 전략이 낮고 저렴하며 집적도가 높아 메모리를 대용량으로 설계하기에 용이

✅ SRAM

- 저장된 데이터가 변하지 않는 RAM
- 전원이 공급되지 않으면 저장된 내용이 소실되는 것은 마찬가지
- DRAM에 비해 속도가 빠르지만 소비 전략이 크고 가격이 비싸 캐시 메모리에서 사용

✅ SDRAM

- 클럭 신호와 동기화된 보다 발전된 형태의 DRAM
- 클럭 신호와 동기화되었다 → 클럭 타이밍에 맞춰 CPU와 정보를 주고받을 수 있다

✅ DDR SDRAM

- 대역폭을 넓혀 속도를 빠르게 만든 SDRAM
- 두 배의 대역폭으로 데이터를 전송

# 캐시 메모리

> CPU의 연산 속도와 메모리 접근 속도의 차이를 줄이기 위해 탄생
> 
- CPU가 아무리 빨라도 메모리에 접근하는 속도가 느리면 효용이 없음
- SRAM 기반의 저장장치
- CPU가 매번 메모리에 왔다 갔다하는 시간이 오래 걸리므로 메모리에서 CPU가 사용할 일부 데이터를 미리 캐시 메모리로 가져옴

![image.png](attachment:7349a017-2c85-454c-9b73-13471f9e2fe0:image.png)

- L1 캐시
    - 코어랑 가장 가까운 캐시 메모리
    - L1l 캐시 : 명령어만 저장
    - L1D 캐시 : 데이터만을 저장하는 캐시
- L2 캐시
    - L2까지 코어 내부에 있음
- L3 캐시
    - 여러 코어가 공유

## 캐시 히트와 캐시 미스

메모리 용량이 작음 → 모든 내용을 가져와 저장할 수 없음

메모리의 일부를 복사하여 저장하는데 , 무엇을 저장해야 할까?

- 캐시 히트
    - 캐시 메모리가 예측하여 저장된 데이터가 실제로 사용되는 경우
- 캐시 미스
    - 예측 실패로 인해 필요한 데이터를 직접 가져와야 하는 경우

## 참조 지역성의 원리

> CPU가 사용할 법한 데이터를 제대로 예측해서 캐시 적중률을 높여야 함
> 
- 시간 지역성
    - 최근에 접근했던 메모리 공간에 다시 접근하려는 경향
    - 변수
- 공간 지역성
    - CPU는 접근한 메모리 공간의 근처에 접근하려는 경향
    - 배열

## 캐시 메모리의 쓰기 정책과 일관성

- 즉시 쓰기
    - 메모리를 항상 최신 상태로 유지
    - 캐시와 메모리 간 일관성이 깨지는 상황을 방지
    - 데이터를 쓸 때마다 참조해야 하므로 버스의 사용 시간과 쓰기 시간이 늘어남
    - 메모리 접근을 최소화하기 위해 캐시 메모리를 만들었는데, 데이터를 쓸 때마다 메모리와 캐시 메모리에 동시에 접근해야 한다면 캐시 메모리를 둔 효율이 떨어짐
- 지연 쓰기
    - 캐시 메모리에만 값을 써 두었다가 추후 수정된 데이터를 한 번에 메모리에 반영하는 방법
    - 쓰기 속도는 빠르지만 메모리와 캐시 메모리 간의 일관성이 깨질 수 있다는 위험
- 때로는 다른 코어가 사용하는 캐시 메모리와의 불일치
    - 다른 코어가 서로 다른 데이터를 대상으로 작업할 수 있기 때문
    

## 캐시는 어떤 상황에 바람직한가?

- 데이터 갱신은 자주 일어나지 않지만, 참조는 빈번하게 일어나는 환경
- DB 조회가 병목이 되어 사용자 경험을 해칠 때

## 어떤 데이터를 캐시에 두어야 하나?

- 자주 찾는 데이터 : 최근 게시물, 인기 검색어 등 빈도가 높은 데이터
- 변경이 드문 데이터
- DB 부하를 유발하는 쿼리 결과

## 읽기 전략

![image.png](attachment:972af71a-2b1d-4d52-a110-7b5c1bdcb1eb:image.png)

⭐**Look-aside → 반복적인 읽기가 많고, 실시간 일관성 엄격 X**

- 장점
    - Redis가 다운되더라도, DB에 요청을 보내면 되기 때문에 서비스가 죽지 않음
    - 실제로 요청된 데이터만 캐시 저장 → 전체 DB를 캐시에 올릴 필요가 없음
- 단점
    - 캐시가 비어있는 경우 모든 요청이 DB로 가서 초기 응답 속도가 느림 → 이를 방지하기 위해 미리 주요 데이터를 캐시에 밀어넣는 작업을 함
    - DB 데이터가 바뀔 때마다 캐시를 지워주지 않으면 사용자는 과거의 데이터를 봄
    - 인기있는 데이터의 TTL이 만료되는 순간, 수많은 요청이 동시에 DB에 요청하는 문제 발생 가능 → 분산 락을 사용하거나, TTL에 무작위 값을 추가해 만료시간을 분산

**⭐Read-Through→ 캐시를 통해서만 데이터를 읽음**

![image.png](attachment:5e013643-27c4-41e3-8dde-f43ab5f666e3:image.png)

- 데이터 동기화를 Server가 아닌 캐시에게 위임하는 방식
- 캐시가 다운될 경우 서비스 이용에 차질
- Replication 또는 Cluster로 구성하여 가용성을 높여아함

## 쓰기 전략

**⭐ Write-Around** 

![image.png](attachment:08211ad9-20f8-4af5-a0a8-f8a08fd139db:image.png)

- 모든 데이터를 데이터베이스에 저장하고, 캐시를 갱신하지 않음
- 쓰기 속도가 가장 빠름
- 캐시 Miss 시만 업데이트

![image.png](attachment:64fe09a2-13b1-421c-bead-2505928a41bf:image.png)

**⭐ Write-through → 일관성을 최우선으로 할 때 사용하는 전략**

- 작동 원리
    - 데이터를 저장하거나 수정할 때, 캐시와 DB 두 곳에 모두 씀(동기적)
    - 먼저 캐시 저장소에 데이터를 저장/업데이트
    - 그 직후, DB에도 동일한 데이터를 저장
    - 캐시와 DB 두 작업이 모두 완료되어야 사용자에게 성공 응답을 보냄
- 장점
    - 데이터 일관성 보장 : DB와 캐시가 무조건 최신 상태로 저장
    - 데이터 유실 위험 없음
    - 데이터가 들어오는 즉시 캐시되므로, 최초 조회 시점에도 캐시 미스가 발생하지 않음
- 단점
    - 쓰기 지연 : 매번 캐시, DB를 업데이트 해야하므로 쓰기 시간이 2배로 소요
    - 리소스 낭비 : 저장만하고 읽지 않아도 무조건 캐시에 올라감
    - TTL을 설정하여 안쓰는 데이터는 삭제해야함

**⭐ Write-back : 캐시와 DB 동기화를 비동기**

![image.png](attachment:aee6aa2a-3cf5-4085-8374-88e043ae627d:image.png)

- 작동 원리
    - 애플리케이션이 캐시에 데이터를 저장하면, DB 저장을 기다리지 않고 즉시 사용자에게 저장 성공 응답을 보냄
    - 특정 주기마다 캐시에 있는 변경된 데이터를 모아서 DB에 일괄 저장
- 장점
    - 읽기 속도가 빠르다
    - DB 부하 감소
- 단점
    - 데이터 유실 위험 : DB 저장되기 전에 캐시 서버가 죽으면, 그 사이 데이터는 영원히 사라짐
    - 구현 복잡 : DB에 없는 데이터를 추적하고 관리하는 로직 필요
    - 데이터 불일치 기간  존재
- 사용 사례
    - 조회수/좋아요 → 100% 정확하지 않아도 되고, 업데이트 빈도가 높은 데이터
    - 로그 수집 → 로그 데이터는 엄청 쌓이지만, 로그 몇 줄이 날아가도 치명적 X
